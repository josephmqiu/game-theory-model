# M1/M2/M3 Refactor Plan — Align Existing Code with Revised Specs

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Refactor the existing ~20K LOC codebase (M1-M3) to support 10 new entity types, extended fields on existing types, new navigation architecture, and the Overview command center.

**Architecture:** The codebase uses a generic CRUD pattern — `EntityTypeMap`, `entityTypes[]`, `STORE_KEY`, and template command types (`AddEntityCommand<T>`) — so adding new entity types is largely mechanical: add to the arrays/maps/schemas, and the generic dispatch/reduce/cascade machinery handles the rest. The bigger refactors are: (1) changing the Assumption type enum (breaking), (2) adding `role` to Player alongside existing `type`, (3) adding `signaling` formalization kind, (4) changing `abstractionLevels` from `coarse/medium/detailed` to `minimal/moderate/detailed`, (5) replacing the view-switcher navigation with sidebar nav, and (6) building the Overview command center.

**Tech Stack:** TypeScript strict, Zod schemas, Zustand store, React 18, React Flow, Tauri 2.0, Vitest

## Chunk 1: M1 — Types and Schemas (data layer)

This chunk adds 10 new entity types, extends 3 existing types, updates enums, and bumps the schema version. It must pass all existing tests plus new ones.

### Key design decision: Generic pattern

The codebase's generic CRUD pattern means most M2 changes come free once M1 types are correct. When you add `'escalation_ladder'` to `entityTypes[]`, the generic `AddEntityCommand<'escalation_ladder'>` type is automatically created. The reducer's `reduceCrudCommand()` already handles any EntityType via `STORE_KEY` lookup. The cascade engine reads `REFERENCE_SCHEMA` declaratively. So the work is: define types → add to registries → add cascade rules → done.
