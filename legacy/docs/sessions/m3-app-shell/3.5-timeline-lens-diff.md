# Session 3.5 — "Timeline, Player Lens, and Diff views work"

## Objective

Build the three remaining analytical views: Timeline (event/model/simulation time), Player Lens (per-player strategic summary with pressure points), and Diff (change tracking from the audit trail). These views complete the visualization layer and make complex multi-game analyses comprehensible.

## Depends on

- Session 3.2 — coordination bus, graph view, inspector panel
- Session 3.1 — Zustand store, app scaffold
- Session 2.3 — `queryEvents` for Diff view audit trail
- Session 1.1 — all entity types

## Contracts

### Types to create

```typescript
// Timeline
interface TimelineEntry {
  id: string
  label: string
  entity_ref: EntityRef
  kind: 'real_world_event' | 'model_move' | 'simulation_turn' | 'evidence_update'
  time_value:
    | { axis: 'event_time'; timestamp: string }
    | { axis: 'model_time'; formalization_id: string; move_index: number }
    | { axis: 'simulation_time'; session_id: string; turn_id: string }
}

// Player Lens
interface PlayerLensState {
  player_id: string
  games_involved: ReadonlyArray<{ game_id: string; role_summary: string }>
  objectives_summary: ReadonlyArray<ObjectiveRef>
  pressure_points: ReadonlyArray<{
    description: string
    source_ref: EntityRef
    severity: 'low' | 'medium' | 'high' | 'critical'
  }>
  cross_game_exposure: ReadonlyArray<{
    link_id: string
    direction: 'incoming' | 'outgoing'
    effect_summary: string
  }>
}

// Diff view
interface DiffEntry {
  entity_ref: EntityRef
  change_type: 'created' | 'updated' | 'deleted' | 'marked_stale' | 'stale_cleared'
  direct_event_id: string
  root_cause_event_id?: string
  timestamp: string
  field_changes?: ReadonlyArray<{ field: string; old_value: unknown; new_value: unknown }>
}
```

### Key functions

```typescript
function buildTimeline(store: CanonicalStore): ReadonlyArray<TimelineEntry>
function buildPlayerLens(playerId: string, store: CanonicalStore): PlayerLensState
function buildDiffEntries(events: PersistedEvent[], store: CanonicalStore): ReadonlyArray<DiffEntry>
```

## Behavior

### Timeline
- Rule: Each TimelineEntry has exactly one time axis. Never mix event time, model time, and simulation time on the same entry.
- Rule: The timeline view displays entries grouped by axis, with separate swimlanes for each.
- Rule: Clicking a timeline entry emits `selection_changed` on the coordination bus.

### Player Lens
- Rule: Pressure points are computed from: high-sensitivity assumptions in the player's games, adverse cross-game link effects, and stale inputs affecting the player's decisions.
- Rule: The lens is read-only. Editing happens in the graph or matrix view.

### Diff view
- Rule: Diff entries are built from persisted SQLite events (session 2.3), not the in-memory EventLog.
- Rule: `root_cause_event_id` tracks cascade causality — if a stale mark was caused by a claim edit, the diff entry for the stale entity links back to the claim edit event.
- Rule: The diff view supports time-range filtering and entity-type filtering.

- Unit test: `buildDiffEntries` from a sequence of persisted events produces correct change types and field diffs.
- Integration test: Timeline entry click emits `selection_changed` and inspector shows the entity.
- Integration test: Diff view shows cascade causality — editing a claim marks an assumption stale, and the diff entry for the assumption links back to the claim edit.

## Guidance

- Timeline: Use a horizontal scrollable layout with vertical lanes per time axis. Do not use a single combined axis.
- Player Lens: This view aggregates data from multiple entity types. Use memoization (`useMemo`) for expensive computations.
- Diff: Query events lazily — only fetch events in the visible time range. Do not load the entire audit trail.
- All three views subscribe to the coordination bus for cross-view synchronization.

### New scope — Phase Detail Screens

This session creates the 10 phase detail screens, accessible via **PHASES > P1 through P10** in the sidebar. Each screen has the standard layout: **sidebar + main content + inspector panel (320px right)**.

Every phase screen displays:
- Phase status badge (pending / active / complete / needs rerun / partial / review needed)
- Pass number if revalidation has fired (e.g., "Pass 2")
- All domain entities are clickable → open inspector

#### Phase 1 — Grounding

7 evidence category cards:
1. **Capabilities & Resources** — military, economic, technological capacity
2. **Economic & Financial** — sanctions, trade flows, financial exposure
3. **Stakeholder Positions** — public statements, diplomatic positions
4. **Impact on Affected Parties** — civilian impact, refugee flows, humanitarian
5. **Timeline** — key dates, deadlines, windows of opportunity
6. **Actions vs Statements** — what players have done vs what they have said
7. **Rules & Constraints** — legal frameworks, treaties, institutional rules

Each card shows: evidence count, key findings summary, status badge (complete/partial/empty). Cards are clickable → inspector shows the evidence items in that category.

#### Phase 2 — Players

Player cards with:
- **Type tag**: Primary / Involuntary / Background / Internal / Gatekeeper
- **Objectives** with AND clauses and priority ordering
- **Stability indicator**: is the objective function shifting?
- **Non-standard utility** section: ideological commitments, honor-based reasoning, audience costs
- **Information asymmetries panel**: what each player knows/doesn't know about others

Each player card is clickable → inspector shows full player profile.

#### Phase 3 — Baseline Model

- **Minimal game selection** with canonical game type match (Prisoner's Dilemma, Chicken, Stag Hunt, etc.)
- **Strategy table** with feasibility annotations per strategy
- **Escalation ladder visualization** — rungs with evidence, reversibility, attribution
- **Move order and time structure** section (event time, model time, simulation time)
- **"What does this model fail to explain?"** — explicit prompt/callout card
- **Cross-game constraint table** (if multiple games): strategies x games matrix with checkmark/cross/question annotations
- **Adjacent games test cards**: "Add only if it changes the answer" decision cards

#### Phase 4 — History

- **Repeated game timeline**: last 5-10 years of player interactions
- **Pattern cards**: defection, tit-for-tat, grim trigger, dual-track deception, etc. Each card shows: pattern definition, historical instances, impact assessment
- **Trust infrastructure assessment gauge**: evidence-backed trust level for each player pair
- **Dynamic inconsistency risks** section: treaty ratification vs executive discretion, electoral cycles
- **Global signaling effects** section: what lessons has the broader system drawn?
- **6-question baseline re-check** from playbook section 4f

#### Phase 5 — Revalidation

This is a **log/dashboard of all revalidation events**, not where revalidation "happens." Revalidation can fire from any phase.

Shows:
- Trigger condition (what changed)
- Source phase (which phase triggered revalidation)
- Target phases (which phases need re-running)
- Resolution status (pending / resolved / in progress)
- Pass number (1st, 2nd, 3rd pass)
- Links to affected entities (clickable → inspector)

#### Phase 6 — Formalization

The most complex phase screen. Sub-sections:

- **6a Formal representations**: embedded matrix/tree views from session 3.4 for each formalization
- **6b Payoff estimation**: `EstimateValue` editors for all payoffs, with rationale and confidence
- **6c Baseline equilibrium**: computed equilibrium results (from L3/derived layer), displayed as read-only until promoted
- **6d Equilibrium selection**: 6 mechanisms panel — path dependence, focal points, relative cost of deviating, commitment devices, institutional rules, salient narratives
- **6e Bargaining dynamics**: outside options, patience/discount factors, deadlines, commitment problems, dynamic inconsistency, issue linkage
- **6f Communication analysis**: cheap talk vs costly signals vs audience costs classification table
- **6g Option value**: "doing nothing may be rational" assessment card
- **6h Behavioral overlays**: prospect theory, cognitive biases — labeled prominently as "ADJACENT — NOT CORE GAME THEORY"
- **6i Cross-game effects**: trigger/effect/magnitude/cascade table for each cross-game link

#### Phase 7 — Assumptions

- **Assumption cards** with sensitivity ratings: critical / high / medium / low
- **Correlated clusters visualization**: which assumptions move together
- **Game-theoretic vs empirical classification** badges
- **Evidence quality flags**: how well-supported is each assumption
- Filter by sensitivity, type, cluster

#### Phase 8 — Elimination

- **Eliminated outcome cards** with phase citations
- Each card shows: outcome description, elimination reasoning (with evidence links), surprise factor badge (how unexpected is this elimination), linked scenarios
- Outcomes grouped by elimination confidence

#### Phase 9 — Scenarios

- **Central thesis card** (falsifiable claim, prominently displayed)
- **Probability bar**: all scenarios summing to ~100%
- **Scenario cards**: narrative, probability with confidence range, key assumptions (clickable), invalidation conditions, model basis (which game drives it, clickable), equilibrium vs discretionary label
- **Tail risks section**: low-probability, high-consequence events
- Cross-game interactions essential to each scenario

#### Phase 10 — Meta-check

- **10 meta-check question cards** with analysis answers
- **6 final test question cards** — prominently displayed in a "Final Test" section
- **Adversarial challenge results**: what would a skeptic say? how does the analysis hold up?

### Top-Level Scenarios Screen (under Scenarios in sidebar nav)

This is the clean **results view** — distinct from Phase 9 which shows the AI's process. Accessible from sidebar nav.

**Layout:** Sidebar + main content + inspector (320px)

**Content:**
- **Central thesis card** (full width, prominent): Falsifiable statement + falsification condition. Clicking → inspector shows full thesis detail with evidence and assumption refs.
- **Probability bar** (full width): All scenarios as colored segments summing to ~100%. Labels below each segment.
- **Scenario cards** (2-4 side by side): Each shows name, narrative summary, probability with confidence range, key assumptions (clickable → inspector), invalidation conditions, model basis (clickable → inspector), "Equilibrium" or "Discretionary" label (prominent, not small badge). Clicking card → inspector shows full scenario detail.
- **Tail risks section**: Warning-styled cards with probability, trigger, consequences, drift assessment. Separate from main scenarios.
- **Eliminated outcomes section**: Compact list with surprise factor badges. Clicking → inspector shows full reasoning with phase citations.
- **Final Test section** (from playbook): 6 question cards, each showing the question and the analysis's answer. Completeness indicator on each (fully/partially/cannot answer).

**User actions:** Adjust probabilities, modify narratives, add scenarios, edit tail risks, export results. All edits dispatch commands through the command spine.

**Hook:** `useScenariosDashboard()` → computed from canonical Scenario, CentralThesis, EliminatedOutcome entities

**File:** `src/components/scenarios/ScenariosScreen.tsx`

### Top-Level Timeline Screen (under Timeline in sidebar nav)

**Layout:** Sidebar + main content + inspector (320px)

**Content:**
- **Filter bar** at top: "All" | "Facts" | "Player Moves" | "Escalation" | "Analysis" (maps to event_type filter)
- **Timeline visualization**: Vertical timeline with events as cards connected to a central line
- **Event types and colors:**
  - Facts (blue) — real-world events from Phase 1
  - Player Moves (amber) — strategic actions from Phase 4
  - Escalation (red) — escalation ladder events from Phase 3
  - Analysis (green) — when AI made key findings, revalidation fired
- Each event card: date, title, description, phase badge, player badges
- Clicking any event → inspector shows full detail with source references and model impact

**Filter hook:** `useTimelineFilters()` → filters by event type, player, game, date range
