# Session 3.4 — "Matrix and tree views with live editing"

## Objective

Build the matrix view for normal-form games and the tree view for extensive-form games. Both views support editing payoffs through the command spine (never direct mutation). Edits in the matrix or tree dispatch `update_payoff` commands, which update the store, trigger graph view updates via the coordination bus, and are undoable. After this session, all three core views (graph, matrix, tree) are live and synchronized.

## Behavior

### Matrix view

- Rule: The matrix view renders when the active formalization is `kind: 'normal_form'`.
- Rule: Rows and columns correspond to the strategy sets of the two primary players. For games with more than 2 players, show a 2-player slice with a player selector to choose which two players to display.
- Rule: Each cell shows the payoff values for all players at that strategy profile. Payoffs are rendered as `EstimateValue` displays (showing value, confidence, and representation type), not bare numbers.
- Rule: Cells are editable. Clicking a payoff cell opens an inline editor for the `EstimateValue`.
- Rule: Saving an edited payoff dispatches an `update_payoff` command through the store's `dispatch` action. The store updates, and the matrix re-renders via Zustand selector.
- Example: Editing the payoff for Player 1 at strategy profile (Cooperate, Defect) dispatches `{ kind: 'update_payoff', payload: { node_id: '<terminal_node_id>', player_id: '<player_1_id>', value: { representation: 'cardinal_estimate', value: 5, confidence: 0.8, rationale: 'Updated estimate', source_claims: [] } } }`.

### Tree view

- Rule: The tree view renders when the active formalization is `kind: 'extensive_form'`.
- Rule: The tree is rendered top-down from the `root_node_id`. Decision nodes show the acting player's name and available actions. Chance nodes show probability labels on outgoing edges. Terminal nodes show payoffs.
- Rule: Information sets are rendered as dashed grouping boxes around nodes that belong to the same set.
- Rule: Terminal payoffs are editable with the same inline editor as the matrix view, dispatching `update_payoff` commands.
- Rule: Edges display their `label` (action name) and, if present, `choice_forecast` or `chance_estimate` as a probability badge.

### Editing through commands

- Rule: All edits dispatch `Command` objects through `store.dispatch()`. No direct mutation of the store or entity objects.
- Rule: After a payoff edit, `isDirty` becomes `true` on the store.
- Rule: Undo reverses the last payoff edit. The matrix/tree re-renders showing the old value.
- Rule: The `EstimateValue` editor must include fields for `representation`, `value`/`min`/`max`/`ordinal_rank`, `confidence`, `rationale`, and `source_claims`. Do not allow saving an estimate without `rationale` and `confidence` (bare numbers are bugs).

### Cross-view synchronization

- Rule: Editing a payoff in the matrix view emits `stale_entities_changed` on the bus if the edit invalidates downstream entities (handled by the integrity reactor).
- Rule: Selecting a cell in the matrix or a node in the tree emits `selection_changed` on the bus. The graph view highlights the corresponding node.
- Rule: Selecting a node in the graph that corresponds to a terminal node scrolls the matrix to the relevant cell or highlights the tree node.
- Rule: Switching to matrix/tree view when a node is already selected in the graph pre-selects the corresponding cell/node.

### View switching

- Rule: The view container automatically shows matrix view for normal-form formalizations and tree view for extensive-form formalizations when the user selects a formalization from the game registry or view switcher.
- Rule: If the active formalization does not match the requested view type (e.g., matrix requested but formalization is extensive-form), show a message: "This formalization is not a normal-form game. Switch to tree view?"

## Guidance

- Use TanStack Table for the matrix layout. It handles column definitions, cell rendering, and accessibility. Do not build a raw HTML table — it will not scale to larger strategy sets.
- The `EstimateValue` editor is shared between matrix and tree views. Extract it as a reusable component from the start. It will also be used in the player lens and scenario views in later milestones.
- For the tree layout, use dagre or a similar algorithm to compute node positions. React Flow can also render tree structures, but a dedicated tree layout component may be cleaner for extensive-form games where the tree is strictly hierarchical.
- The `update_payoff` command targets a node and player. For the matrix view, you need to map (row strategy, column strategy) back to the corresponding terminal node ID. This mapping lives in the `NormalFormCell` data — each cell knows which strategy profile it represents and which payoffs it contains.
- Do not implement full payoff matrix editing (adding/removing strategies or players) in this session. That requires structural commands that change the formalization shape. This session handles payoff value editing only.
- Be cautious with re-renders. The matrix can have many cells, and the tree can have many nodes. Use narrow selectors: each `PayoffCell` should subscribe to its own payoff value, not the entire formalization.
- Information sets in the tree view are visual groupings only in this session. Do not build information-set editing or the game-theoretic implications (they affect solver logic in M4).

## Revision Notes (Step 2)

> Step 2 repositions matrix and tree views as Phase 6 (Formalization) detail components rather than top-level views.

### Primary home: Phase 6 detail screen

- Matrix and tree views serve the **Phase 6 (Formalization) detail screen** primarily. When a user navigates to PHASES > P6 in the sidebar, the formalization sub-sections (6a formal representations) embed the matrix and tree views for each formalization.
- They are also accessible from the **Game Map** (MODELS > Game Map): selecting a specific formalization node opens the matrix or tree view inline or in a dedicated panel, depending on the formalization kind.

### Inspector panel integration

- Clicking a **cell in the matrix** opens the inspector panel with payoff/strategy detail for that cell: all player payoffs as `EstimateValue` displays, the strategy profile, and the terminal node reference.
- Clicking a **node in the tree** opens the inspector panel with that node's detail: acting player, available actions, information set membership, payoffs (for terminal nodes).
- Inspector shows "Open in Phase 6" link to navigate to the Formalization phase detail screen.

### View-switcher removal

- Matrix and tree views no longer appear as top-level tabs in a view-switcher. They are embedded within the Phase 6 screen and accessible via the Game Map inspector.
- The `view_requested` coordination event for matrix/tree now navigates to Phase 6 with the formalization pre-selected.

### Retained behavior

- All editing capabilities are **unchanged**: inline `EstimateEditor`, `update_payoff` commands through the command spine, undo/redo.
- Cross-view synchronization via the coordination bus is retained.
- `useNormalFormViewModel` and `useExtensiveFormViewModel` hooks are unchanged.
- Information set grouping in the tree view is retained.
