# Session 3.7 Б─■ Overview Command Center

## Goal

The Overview screen is the primary workspace Б─■ a conversation-first command center where users can stay for the entire analysis. Users ask questions, give direction, and receive structured results inline. The AI conducts the full game theory workflow per the playbook; the user watches, steers, and gets results without navigating away.

### Main Content Components

#### ConversationArea

Vertical scrollable area with messages. Two message types:

- **UserMessage**: right-aligned, subtle accent tint (`#F5A62315` fill, `#F5A62340` border), plain text. Font: JetBrains Mono body.
- **AIMessage**: left-aligned, card styling (`#141415` fill, `#27272A` border). Structured content rendered as rich components.

AI message sub-types (distinguished by `message_type` field):

| Sub-type | Visual treatment | Content |
|---|---|---|
| `phase_transition` | Amber left border + phase badge | "Starting Phase 2: Player Identification..." |
| `finding` | Standard card | Structured content with inline entity references (clickable Б├▓ inspector) |
| `proposal` | Standard card with action buttons | Inline `ProposalCard` components with accept/reject/modify buttons |
| `result` | Green left border + completion badge | Inline scenario/thesis cards |
| `revalidation` | Warning left border + caution badge | "Phase 4 broke the baseline. Looping back to Phase 2." |
| `steering_ack` | Subtle italic | Acknowledgment that user direction was received and applied |

All entity references in AI messages are clickable Б├▓ open inspector with that entity's detail.

#### ConversationInput

Fixed at the bottom of the conversation area.

- Text input: `#141415` fill, `#27272A` border, `cornerRadius` 8, height 40
- Placeholder: "Ask a question, give direction, or request follow-up..."
- Send button: `#F5A623` fill, send icon, 32x32
- Dispatches user message to conversation store and AI orchestrator
- Keyboard: Enter to send, Shift+Enter for newline

#### PhaseProgressBar

Compact horizontal bar below the conversation showing all 10 phases.

- Small circles connected by lines, one per phase
- Colors: green (complete), amber pulsing (active), orange (needs rerun / revalidation), gray (pending)
- Shows pass number indicator if revalidation has fired (e.g., "Pass 2" label above the circle)
- Each phase circle is clickable Б├▓ navigates to the phase detail screen via sidebar routing

#### KeyFindingsDashboard

Horizontal scrollable row of compact summary cards below the progress bar.

- Cards update live as analysis progresses
- Example cards: "12 evidence items" | "6 players" | "3 games" | "18 assumptions" | "4 eliminated" | "4 scenarios"
- Each card is clickable Б├▓ navigates to the relevant screen (e.g., "6 players" Б├▓ MODELS > Players) OR opens the inspector with a summary
- Card styling: `#0F0F10` fill, `1px #27272A` border, compact padding

### Conversation Store

```typescript
interface ConversationMessage {
  id: string
  role: 'user' | 'ai'
  timestamp: string
  content: string              // plain text for user, markdown for AI
  structured_content?: StructuredContent  // for AI messages only
  phase?: number               // which phase this message relates to
  message_type?: 'phase_transition' | 'finding' | 'proposal' | 'result' | 'revalidation' | 'steering_ack'
}

interface StructuredContent {
  proposals?: ProposalGroup[]
  entity_refs?: EntityRef[]    // clickable entity references in the message
  scenario_cards?: EntityRef[]
  thesis?: EntityRef
  findings_summary?: ReadonlyArray<{
    label: string
    count: number
    entity_type: EntityType
  }>
}

interface ProposalGroup {
  id: string
  phase: number
  proposals: ReadonlyArray<Proposal>
  status: 'pending' | 'partially_accepted' | 'accepted' | 'rejected'
}

interface Proposal {
  id: string
  description: string
  entity_previews: EntityPreview[]  // reuse EntityPreview from session 5.3
  status: 'pending' | 'accepted' | 'rejected' | 'modified'
}
```

**Storage layer:**
- Conversation messages are stored in **L2 (view state)** during the active session.
- Persisted to **SQLite** for session continuity across app restarts. Conversation is NOT stored in `.gta.json` Б─■ it is not canonical data.
- Proposals reference the diff review mechanism from M5 (session 5.3).

### Key Behaviors

1. **User types message** Б├▓ dispatched to conversation store Б├▓ forwarded to AI orchestrator Б├▓ AI processes and responds with structured messages.
2. **AI proposals appear inline** Б├▓ user accepts/rejects/modifies individual proposals without leaving the Overview. Accepted proposals dispatch commands through the command spine.
3. **Phase transitions** shown as special messages with amber left border and phase badge. The phase progress bar updates simultaneously.
4. **Entity references** in AI messages are clickable Б├▓ inspector shows the entity's detail.
5. **Key findings dashboard** auto-updates as entities are added to the canonical store (driven by Zustand selectors on entity counts).
6. **Phase progress bar** reflects the current pipeline state, updated by the AI orchestrator or manual phase completion.
7. **Steering**: user can type directions at any time during the analysis. The AI acknowledges with a `steering_ack` message and adjusts current phase behavior accordingly.
8. **Scroll behavior**: new messages auto-scroll to bottom unless the user has scrolled up. A "scroll to bottom" button appears when the user is not at the latest message.

### New Analysis Screen

This screen is shown when the user clicks "New Analysis" from the Welcome screen. It is a pre-analysis entry point Б─■ no sidebar, centered layout.

**Layout:** Centered content, no sidebar, no inspector (pre-analysis state)

**Components:**

- Optional: "Attach reference documents" file drop zone

- Two action buttons:
  - "Begin Analysis" (primary, amber) Б├▓ creates AnalysisFile, navigates to Overview, starts Phase 1
  - "Manual Mode" (secondary) Б├▓ creates AnalysisFile, navigates to Overview with conversation disabled

**Hook:** `useNewAnalysis()` Б├▓ `{ description, attachments, startAnalysis(), startManual() }`

**Route:** `/new-analysis` (ViewType: `new_analysis`)

## Behavior rules

- Conversation **persists across app restarts** via SQLite. Re-opening an analysis restores the conversation history.
- **Empty state**: a welcome message explaining how to start Б─■ "Describe a situation to analyze, or open an existing analysis." No phase progress bar or key findings shown until analysis begins.
- **No MCP client connected**: conversation input shows a message explaining that an MCP-capable AI client (Claude Desktop, ChatGPT) can connect to drive analysis, or the user can use manual mode via the phase screens. The phase progress bar and key findings dashboard still render if entities exist from manual entry.
- **Manual mode**: conversation area is hidden entirely. The Overview shows only the phase progress bar and key findings dashboard, acting as a status hub. Users navigate to phase screens via the sidebar to do their work.
- **Proposal lifecycle**: proposals start as `pending`. User can accept (dispatches command), reject (marks as rejected, no command), or modify (opens editor, then dispatches modified command). A `ProposalGroup` is `partially_accepted` if some proposals are accepted and others are still pending or rejected.
- **Revalidation messages**: when revalidation fires, a `revalidation` message appears in the conversation with the trigger condition, affected phases, and what the AI plans to re-run. The phase progress bar updates to show warning icons on affected phases.

## Guidance

- The conversation area is the centerpiece. Optimize for readability Б─■ adequate spacing between messages, clear visual distinction between user and AI messages.
- AI messages can be long and structured. Use collapsible sections within AI messages for verbose content (e.g., collapse detailed evidence lists, expand on click).
- The phase progress bar should be compact Б─■ one line, not a full-width section. It is supplementary to the conversation, not competing with it.
- Proposal accept/reject should be fast. Single click to accept, single click to reject. "Modify" opens an inline editor (reuse `EntityEditor` from session 3.6) that pre-fills with the proposal's values.
- Do not build the AI orchestrator in this session. The conversation store and UI are built and tested against mocked AI responses. The actual AI pipeline comes in M5/M6.
- Use the design system: dark theme, `#F5A623` amber accent, JetBrains Mono for body text, Space Grotesk for titles. Card style: `#0F0F10` fill, `1px #27272A` border.
- Conversation messages should render markdown in AI responses (use a lightweight markdown renderer). User messages are plain text only.
- The `InlineProposalCard` component should reuse the `ProposalCard` / `EntityPreview` components from session 5.3 if they exist. If not (session 5.3 may not be built yet), create a standalone version with the same interface so it can be swapped later.
