# Session 3.1 — "The app launches and loads a model"

## Objective

Scaffold the Tauri desktop app with React, Vite, and TypeScript strict mode. Create the Zustand store that wraps the `CanonicalStore` from M1 and the `EventLog` from M2, wire up the `dispatch` function so it can be called from React, and load a `.gta.json` fixture file on launch. After this session, the app window opens, a fixture model is loaded into the store, and commands can be dispatched from the browser dev console.

## Behavior

### Store initialization

- Rule: The store initializes with an empty `CanonicalStore`, an empty `EventLog` (cursor 0, no events), an empty `InverseIndex`, and `ViewState` defaulting to `{ activeGameId: null, activeFormalizationId: null, inspectedRefs: [], activeView: 'graph' }`.
- Rule: `loadFile` runs the full load pipeline from session 1.2, populates the canonical store, builds the inverse index, sets the file path and analysis name, and resets the event log.

### Dispatch integration

- Rule: The store's `dispatch` action calls the pure `dispatch` function from M2 with the current canonical store and event log.
- Rule: If the result is `committed`, the store updates `canonical`, `eventLog`, `inverseIndex`, and sets `isDirty: true`.
- Rule: If the result is `rejected`, the store is not modified. The result is returned to the caller.
- Rule: If `dryRun` is true, the store is not modified. The result is returned for preview.

### Undo/redo integration

- Rule: `undo()` calls the pure `undo` function with current state. If it returns a new state, the store updates. Returns `true` if successful, `false` if nothing to undo.
- Rule: `redo()` works the same way.

### Save integration

- Rule: `saveFile` converts the canonical store to an `AnalysisFile` and calls `saveAnalysis`. On success, sets `isDirty: false`.
- Rule: If `filePath` is null, `saveFile` should throw (no path to save to). A "Save As" flow is out of scope for this session.
- Rule: If the load pipeline (from session 1.2) returns a failure result, the app should render a recovery mode view: read-only display of the raw JSON, the specific error that occurred, and an option to export individual entities as JSON snippets. This does not need to be polished — a basic code editor (Monaco/CodeMirror) showing the raw file with error annotations is sufficient.

### App shell

- Rule: The app has a single window with a view switcher (tabs or dropdown for `ViewType`), a placeholder content area, and a toolbar with undo/redo/save buttons.
- Rule: On launch, the app loads a hardcoded fixture file path (or prompts for one via Tauri file dialog).
- Rule: Views are placeholder components that render the view name. Real view implementations come in sessions 3.2-3.4.
- Rule: The sidebar navigator shows games as top-level items with formalizations nested underneath. Selecting a formalization sets `activeFormalizationId` and the main view area shows the appropriate view for that formalization's kind (matrix for normal-form, tree for extensive-form, graph for general).

## Guidance

- Use `zustand` with the `immer` middleware only if you need it for the store actions themselves. The core `dispatch` function from M2 is already pure and immutable — the store action just needs to call it and `setState` with the result.
- The Tauri scaffold should use `create-tauri-app` conventions. Pin Tauri 2.0.x. Use `@tauri-apps/api` for filesystem access.
- TypeScript strict mode means `"strict": true` in tsconfig. No `any` types. This is non-negotiable.
- The fixture file is critical for all subsequent view sessions. Make it realistic: at least 2 games, 2 players, 1 formalization, several nodes/edges, a few evidence entities, and at least one entity with a stale marker. Do not use trivial single-entity fixtures.
- Expose the store on `window.__store` in development mode for console-driven testing. Remove or guard this behind `import.meta.env.DEV` for production.
- Do not spend time on styling or layout polish. The app shell needs to work, not look good. A plain HTML structure with functional buttons is sufficient. Visual design comes later.
- Do not build the coordination bus yet. That is session 3.2's concern. View switching just swaps the rendered component.
- The app shell should use a persistent navigation sidebar (not just tabs/dropdown) for the view system. The sidebar shows: (1) a game/formalization tree navigator at the top, (2) view type switches below, (3) a collapsible inspector panel on the right. This scales better than a flat tab bar as the view count grows from 4 to 9+. Refer to Fluent 2 Navigation patterns for desktop IA guidance.
- All views must be keyboard-navigable. Use semantic HTML, ARIA labels on interactive elements, and ensure focus management when switching views. This is a cross-cutting concern that starts here and must be maintained in every subsequent UI session.
- The spec's directory structure shows a GameRegistry panel. For now, game management happens through the inspector panel (click a game in the graph, see its details, edit inline). A dedicated GameRegistry panel can be added later if the inspector approach proves insufficient.

## Revision Notes (Step 2)

> **Implementation note:** The revision notes below supersede the original contract where they conflict. Specifically: the original `ViewType` enum (9 values), `setActiveView`, and view-switcher architecture are replaced by sidebar navigation with an expanded `ViewType` (~24 values) and route-based navigation. The original `AnalysisStoreState` shape is preserved but extended with `inspectorState` and `phaseStatuses`.

> Step 2 revises the app shell from a view-switcher to a sidebar-navigation architecture with an inspector panel on every screen. The source of truth for the new navigation structure is `docs/ux-journey-map.md`.

### Navigation architecture change

- **Replace `activeView` view-switcher with sidebar navigation component.** The flat tab/dropdown view-switcher is removed. Navigation is now a persistent sidebar with hierarchical sections.
- Sidebar structure matches the UX journey map:
  ```
  Overview          ← default landing screen (command center)
  Timeline
  Scenarios
  Play-outs

  MODELS            ← section header
    Game Map
    Evidence
    Players
    Assumptions

  PHASES            ← collapsible section
    P1  Grounding        [status icon]
    P2  Players          [status icon]
    P3  Baseline Model   [status icon]
    P4  History          [status icon]
    P5  Revalidation     [status icon]
    P6  Formalization    [status icon]
    P7  Assumptions      [status icon]
    P8  Elimination      [status icon]
    P9  Scenarios        [status icon]
    P10 Meta-check       [status icon]

  ────────
  Settings
  ```
- The `ViewType` enum expands to cover all routes: `overview`, `timeline`, `scenarios`, `playout`, `game_map`, `evidence`, `players`, `assumptions`, `phase_1` through `phase_10`, `settings`, `welcome`, `new_analysis`.
- Sidebar shows phase status icons per the UX journey map: `○` pending, `◉` active, `✓` complete, `⚠` needs rerun, `◐` partial, `●` review needed.
- PHASES section is **collapsible** (collapsed by default for new users).

### Inspector pattern (new)

- Every screen (except Welcome and New Analysis) has a **320px right-side inspector panel**.
- Inspector shows entity detail when the user clicks a domain entity in the main content area.
- `InspectorState` added to view state:
  ```typescript
  interface InspectorState {
    inspectedEntity: EntityRef | null
    breadcrumb: string[]  // e.g. ["Player", "United States"]
  }
  ```
- Inspector header: breadcrumb trail + entity type badge + "Open in Phase X" link to navigate to the phase where the entity was produced.
- Inspector content: **read-only by default**. An "Edit" button enters edit mode and dispatches commands through the command spine.
- **Same entity shows the same inspector view** regardless of which screen it is viewed from. Inspector rendering is entity-type-driven, not screen-driven.

### Inspector Entity Renderers

Each entity type has a dedicated renderer component. When `inspectedEntity` changes, the inspector selects the appropriate renderer.

| Entity Type | Inspector shows | Component |
|---|---|---|
| Player | Role tag, type, objectives (AND clauses), priority ordering, stability, non-standard utility, games they participate in, trust levels, information state | `InspectorPlayer.tsx` |
| Game (StrategicGame) | Canonical type, players, formalization types, equilibrium results, cross-game links, escalation ladder | `InspectorGame.tsx` |
| Source/Observation/Claim/Inference | Full derivation chain (source → observation → claim → inference → assumption), which assumptions depend on it, phase attribution | `InspectorEvidence.tsx` |
| Assumption | Sensitivity rating, type (behavioral/capability/etc.), "what if wrong?", correlated cluster, dependent scenarios, evidence chain, game-theoretic vs empirical | `InspectorAssumption.tsx` |
| Scenario | Full narrative, probability with range, all key assumptions (clickable), invalidation conditions, model basis, cross-game interactions, equilibrium vs discretionary label | `InspectorScenario.tsx` |
| EliminatedOutcome | Full reasoning chain with phase citations, evidence refs, surprise factor, related scenarios | `InspectorEliminatedOutcome.tsx` |
| EscalationLadder/Rung | All rungs with climbed/not-climbed status, reversibility, player attribution, escalation dominance, stability-instability paradox | `InspectorEscalation.tsx` |
| TrustAssessment | Trust level gauge, evidence, interaction history, driving patterns, implications for cooperation | `InspectorTrust.tsx` |
| CrossGameLink | Trigger, effect type, magnitude, direction, cascade analysis, source/target games | `InspectorCrossGameLink.tsx` |
| Formalization | Kind, purpose, abstraction level, equilibrium results, sensitivity analysis, assumptions introduced | `InspectorFormalization.tsx` |
| SignalClassification | Classification (cheap talk/costly/audience cost), cost description, informativeness with conditions | `InspectorSignal.tsx` |
| CentralThesis | Statement, falsification condition, evidence refs, assumption refs, forecast basis | `InspectorThesis.tsx` |

Each renderer includes:
- Header: entity type badge + entity name/label
- Breadcrumb: "Player > United States" or "Assumption > Iran retains stockpile"
- "Open in Phase X" link (navigates to the phase detail screen where this entity lives)
- "Edit" button (enters edit mode, dispatches update_X commands on save)
- Collapsible sections for different detail categories

**Not clickable** (display only): Section headers, status badges, probability bars, progress indicators, UI chrome.

### Zustand store changes

- `viewState.activeView` is **superseded** by route-based navigation (React Router or a lightweight equivalent). The sidebar drives route changes; `activeView` becomes a derived value from the current route if needed for backward compatibility.
- Add `viewState.inspectorState: InspectorState` for the inspector panel.
- Add `viewState.phaseStatuses: Record<number, PhaseStatus>` for sidebar status icons, where `PhaseStatus` is `'pending' | 'active' | 'complete' | 'needs_rerun' | 'partial' | 'review_needed'`.
- Keep `viewState.activeGameId` and `viewState.activeFormalizationId` — these are still needed for model lens screens.
