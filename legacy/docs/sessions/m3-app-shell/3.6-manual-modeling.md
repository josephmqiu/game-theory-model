# Session 3.6 — "Users can build a complete model by hand"

## Objective

Build the manual modeling workflow so users can create, edit, and connect all canonical entity types through the UI without AI assistance. The app must be a capable strategic modeling tool even when no AI provider is configured. This session adds creation forms, entity editors, and relationship linkers for games, players, formalizations, assumptions, contradictions, cross-game links, scenarios, and playbooks.

## Behavior

### Entity creation
- Rule: Every canonical entity type has a creation form accessible from the toolbar or context menu.
- Rule: Creation forms use Zod schemas for input validation. Invalid forms show inline errors and cannot submit.
- Rule: Forms dispatch `add_*` commands through the command spine. No direct store mutation.
- Rule: After creation, the new entity is selected in the inspector and the relevant view scrolls to it.

### Relationship linking
- Rule: Cross-game links are created via a "Link Games" panel that shows two game selectors, a trigger selector, and an effect configuration form.
- Rule: Assumptions can be attached to games, formalizations, nodes, and edges via an "Add Assumption" action in the inspector context menu.
- Rule: Contradictions are created by selecting two claims and choosing "Mark as Contradicting."

### Inline editing
- Rule: Entity fields are editable inline in the inspector panel. Changes dispatch `update_*` commands.
- Rule: The evidence notebook (session 3.3) gains edit capability: sources, observations, and claims can be created and edited manually, not just viewed.

### Empty state
- Rule: When no analysis is loaded, the app shows a welcome screen with: "Create New Analysis", "Open File", and "Load Example" options.
- Rule: A new analysis starts with an empty CanonicalStore and prompts the user to create their first game.

## Guidance

- Forms should be simple and focused. Do not overload a single form with all entity fields. Start with required fields; optional fields expand on demand.
- Use Zod schemas from session 1.1 for validation. Do not duplicate validation logic.
- The welcome screen should be visually clean and not overwhelming. Three options maximum.
- This session makes the evidence notebook (3.3) editable. Extend its components rather than rewriting them.
- Manual creation should feel fast. Optimize for keyboard navigation: Tab between fields, Enter to submit, Escape to cancel.
