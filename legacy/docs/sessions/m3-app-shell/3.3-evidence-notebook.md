# Session 3.3 — "Evidence is browsable and views cross-link"

## Objective

Build the evidence notebook view that displays the full evidence ladder (sources, observations, claims, inferences, assumptions, contradictions, derivations) in a browsable, structured layout. Wire it to the coordination bus so that selecting an evidence entity in the notebook highlights referencing nodes in the graph view, and clicking an evidence link in the graph inspector navigates to the notebook. After this session, the evidence layer is visible, browsable, and synchronized with the graph.

## Behavior

### Evidence notebook layout

- Rule: The notebook displays evidence entities grouped by type in the evidence ladder order: Sources at the top, then Observations, Claims, Inferences, Assumptions, Contradictions.
- Rule: Each entity renders as a card showing its key fields: ID, statement/text/description, confidence (where applicable), and a stale badge if stale.
- Rule: Each card shows clickable links to its referenced entities (e.g., a Claim shows links to its `based_on` Observations, and Inferences show links to their `derived_from` Claims).
- Rule: Clicking a link scrolls to and highlights the target entity within the notebook, or switches to the appropriate view if the target is a non-evidence entity (e.g., a GameNode).

### Derivation chain display

- Rule: When an evidence entity is selected, its derivation chain is visualized: upstream (what it derives from) and downstream (what derives from it).
- Rule: This uses `DerivationEdge` entities to trace the chain. The `relation` field is displayed as the edge label (supports, contradicts, infers, etc.).
- Rule: The chain display is a compact inline visualization within the notebook, not a separate graph view.

### Cross-view synchronization

- Rule: Selecting a `Claim` in the notebook emits `selection_changed` on the coordination bus.
- Rule: The graph view receives this event and highlights any `GameNode` or `GameEdge` whose `claims`, `source_claims`, or `assumptions` arrays reference the selected claim.
- Rule: The highlighting is done via the bus `highlight_dependents` event, not by modifying the store.
- Rule: Clicking an evidence link in the graph inspector panel emits `view_requested` with `view: 'evidence_notebook'` and `context` pointing to the evidence entity, causing the app to switch to the notebook view and scroll to that entity.

### Stale badges

- Rule: Stale entities in the notebook show an amber badge with the stale reason tooltip.
- Rule: The badge is driven by `useIsStale`, same as in the graph view.
- Rule: When stale entities change, the bus fires `stale_entities_changed` for a transient flash animation. This is purely visual and ephemeral.

### Contradiction display

- Rule: Contradictions are rendered as paired cards showing both the `left_ref` and `right_ref` claims side by side, with the `resolution_status` and `description`.
- Rule: Contradictions with status `'open'` are visually prominent (e.g., a red/orange border).

### Uncertainty components

- Rule: These uncertainty components are reusable across views. `ConfidenceBadge` appears wherever `EstimateValue` is displayed. `AssumptionList` appears in the inspector for any entity with assumptions. `DependencyPanel` uses the inverse reference index from session 2.2.

## Guidance

- The evidence notebook is not just a flat list. The visual grouping by evidence type (sources -> observations -> claims -> inferences -> assumptions) reflects the epistemic ladder. Make the hierarchy visually clear with section headers or indentation.
- Use narrow Zustand selectors. `useEvidenceLadder` should select only the evidence-related maps from the store. Do not select the entire `CanonicalStore` and filter in the component.
- `useDerivationChain` walks `DerivationEdge` entities, which have `from_ref` and `to_ref` fields. These are string IDs, not `EntityRef` objects. You need to resolve them to `EntityRef` by finding the entity type (derivation edges connect across types). Use the inverse index or scan by ID prefix.
- The cross-view highlight via `highlight_dependents` should be transient. When the user clicks elsewhere, the highlight clears. Use `highlight_clear` to reset.
- Do not build inline editing of evidence entities in this session. The notebook is read-only for now. Editing goes through commands dispatched from the inspector panel or a future edit modal.
- Contradictions are a first-class concept, not a footnote. Give them visual prominence. Open contradictions are signals that the model needs attention.
- The derivation chain visualization can be simple — a vertical list of nodes connected by labeled arrows. Do not build a full graph renderer for this. Keep it compact enough to fit in a sidebar or inline panel.

- The Evidence screen is accessed via **MODELS > Evidence** in the sidebar, not through a view-switcher tab.

- Each evidence item in the notebook displays a **phase attribution badge** showing which phase produced or last modified it (e.g., "P1", "P4", "P6").
- Badge color follows the phase's status color from the sidebar.
- This helps users understand evidence provenance at a glance without opening the inspector.

- Evidence ladder visualization (Sources → Observations → Claims → Inferences → Assumptions → Contradictions) is **unchanged**.
- Cross-view synchronization via the coordination bus is retained.
- Stale badges, contradiction paired cards, confidence badges all retained.
