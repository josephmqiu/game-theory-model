## Contracts

### Types to create

```typescript
interface BaseCoordinationEvent {
  correlation_id: string
  source_view: string
}

type CoordinationEvent = BaseCoordinationEvent & (
  | { kind: 'selection_changed'; refs: EntityRef[] }
  | { kind: 'focus_entity'; ref: EntityRef }
  | { kind: 'scroll_to'; ref: EntityRef }
  | { kind: 'highlight_dependents'; root: EntityRef; depth: number }
  | { kind: 'highlight_clear' }
  | { kind: 'view_requested'; view: ViewType; context?: EntityRef }
  | { kind: 'stale_entities_changed'; refs: EntityRef[] }
)

interface CoordinationSnapshot {
  selection: ReadonlyArray<EntityRef>
  focusedEntity: EntityRef | null
  highlightedDependents: { root: EntityRef; depth: number } | null
}

class CoordinationBus {
  emit(event: CoordinationEvent): void
  subscribe(kind: CoordinationEvent['kind'], handler: (event: CoordinationEvent) => void): () => void
  getSnapshot(): Readonly<CoordinationSnapshot>
}
```

## Behavior

### Graph rendering

- Rule: The graph view renders all entities for the active game's formalizations as React Flow nodes.
- Rule: `GameNode` entities render as custom nodes with type-specific styling: decision nodes (square), chance nodes (diamond/circle), terminal nodes (rounded rectangle).
- Rule: `GameEdge` entities render as custom edges with labels showing the action/move.
- Rule: Players are rendered as group nodes or header labels that visually cluster their decision nodes.
- Rule: Stale entities display a warning badge overlay on their node. The badge is driven by `useIsStale`, not the coordination bus.
- Rule: The graph zooms to fit the loaded model on initial render.
- Rule: `AssumptionNode` renders assumptions as distinct visual elements (different shape/color from decision/chance/terminal nodes) with sensitivity indicators.
- Rule: `CrossGameEdge` renders cross-game links with a distinct style (dashed, different color) showing the effect type.
- Rule: `DerivationEdge` renders evidence-to-model relationships showing the derivation relation type.

### Selection three-tier model

- Rule: Clicking a node updates all three tiers in sequence:
  1. **Coordination bus** (immediate): Emits `selection_changed` with the clicked entity's ref.
  2. **Zustand store** (next tick): Updates `viewState.inspectedRefs` with the selected entity.
  3. **URL** (debounced): Updates search params to encode the selection for deep-linking.
- Rule: On mount, a view reads `inspectedRefs` from the Zustand store (hydrated from URL on page load), then calls `bus.getSnapshot()` for ephemeral highlight state.

### Coordination bus

- Rule: The bus is a singleton, created once at app initialization.
- Rule: Views use `correlation_id` to prevent echo loops: a view ignores events with its own `source_view`.
- Rule: The bus maintains a `CoordinationSnapshot` that newly-mounted views can read to sync up with current selection state.
- Rule: The bus fires after store commit. When `dispatch` completes, the store updates first, then the bus emits relevant signals.

### Inspector panel

- Rule: When `inspectedRefs` has one or more entries, an inspector panel shows the details of the first inspected entity.
- Rule: The inspector renders a read-only view of the entity's fields, including its type, ID, and all domain-specific fields.
- Rule: For entities with evidence references (claims, source_claims, assumptions), the inspector shows clickable links that update the selection to that entity.

### Feedback loop prevention

- Rule: `useCoordinationHandler` checks `event.source_view` and `correlation_id` to skip events originating from the same view or already-processed events.

## Guidance

- Use React Flow v11+ with custom node types. Register custom nodes via `nodeTypes` prop, not inline render functions. This avoids re-render issues.
- The graph layout algorithm (dagre, elk, or manual positioning) is a secondary concern. Start with dagre for automatic layout. Users will be able to manually reposition nodes later.
- The coordination bus is NOT a React context or a Zustand store. It is a plain event emitter with a lightweight snapshot. Do not over-engineer it. A class with `Map<kind, Set<handler>>` and a `CoordinationSnapshot` object is sufficient.
- The inspector panel should be a sidebar, not a modal. It should stay open while the user interacts with the graph.
- Do not try to render all entity types as graph nodes. The graph shows `GameNode` and `GameEdge` entities from the active formalization. Evidence entities (sources, claims, etc.) appear in the inspector panel and in the evidence notebook (session 3.3), not as graph nodes.
- React Flow handles zoom and pan natively. Do not build custom zoom controls unless React Flow's defaults are insufficient.
- The URL sync for deep-linking can use `URLSearchParams` directly. Do not add a routing library just for this. Keep it simple: `?inspect=game_node:node_abc&view=graph`.
- The stale badge on nodes is data-driven (from the store), not bus-driven. The bus `stale_entities_changed` event is only for transient UX effects (amber flash). Do not confuse the two.
