## Behavior

### Load pipeline (8 steps)

- Rule: Step 1 (Parse) — Read raw JSON, verify it is an object with a `schema_version` field. If not, reject with a parse error.
- Rule: Step 2 (Migrate) — If `schema_version < current`, run the migration pipeline. (In this session, current is `1`, so migration is a no-op pass-through. Session 1.3 adds the real pipeline.)
- Rule: Step 3 (Validate) — Run the `AnalysisFile` Zod schema. If validation fails, return the Zod errors.
- Rule: Step 4 (Structural checks) — Before converting arrays to maps: verify all entities have non-empty `id` fields, verify no duplicate IDs within any single array, verify no ID collisions across entity types.
- Rule: Step 5 (Convert) — Transform arrays to ID-keyed maps using `STORE_KEY`.
- Rule: Step 6 (Build derived structures) — Placeholder for inverse reference index (built in session 2.2).
- Rule: Step 7 (Integrity check) — Placeholder for post-validation invariants (built in session 2.2).
- Rule: Step 8 (Initialize event log) — Return a fresh cursor at 0. (Event log is built in session 2.1.)

- Rule: If any step fails, the file should open in recovery mode. For this session, "recovery mode" means returning the error and the raw JSON for inspection.

### Array-map conversion

- Rule: `arrayToMap` must reject arrays containing duplicate IDs. Throw a specific error naming the duplicate.
- Rule: `mapToArray` must produce a deterministic order. Sort by `id` lexicographically.
- Rule: Round-trip invariant: `mapToArray(arrayToMap(arr))` deep-equals `arr` when `arr` is sorted by `id` and has no duplicates.

### Atomic save

- Rule: Write to a `.tmp.<timestamp>` file first.
- Rule: Re-parse and validate the temp file with the current Zod schema before renaming.
- Rule: If verification fails, delete the temp file and throw. The original file is untouched.
- Rule: If the target file already exists, copy it to `.backup` before renaming the temp file over it.
- Example: Saving to `analysis.gta.json` writes to `analysis.gta.json.tmp.1710432000000`, verifies, copies `analysis.gta.json` to `analysis.gta.json.backup`, then renames the temp file.

### Stale marker preservation

- Rule: `stale_markers` arrays on entities are serialized into the `.gta.json` file inline on each entity object. No separate stale-tracking structure.
- Rule: On load, `stale_markers` arrays are preserved as-is through the array-to-map conversion.

## Guidance

- The `.gta.json` format uses arrays for human readability. The in-memory store uses maps for patch stability (JSON Patch on arrays is fragile because indices shift on insert/delete). If you find yourself indexing into arrays at runtime, something is wrong.
- `mapToArray` sorts by ID to ensure deterministic output. Without this, file diffs would be noisy.
- The load pipeline steps 6-8 are intentional placeholders. Return stub values. Sessions 2.1 and 2.2 will fill them in. Do not skip the steps — the pipeline structure must exist so later sessions can slot in.
- For atomic save, use Tauri's filesystem API in production but design the functions to accept injected I/O functions for testability. Tests should use an in-memory filesystem mock, not real disk I/O.
- Cross-type ID collision checking works because IDs are prefixed (`claim_abc`, `asmp_def`). But the check must still run — a bug in ID generation could produce collisions.
- Do not import or depend on command/event types. This layer is purely about data shape and I/O.
