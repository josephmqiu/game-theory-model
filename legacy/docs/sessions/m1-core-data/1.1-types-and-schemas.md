# Session 1.1 — "All types compile, all schemas validate"

## Objective

Define every TypeScript interface and Zod schema that the canonical data model requires. This is the foundation session — nothing exists yet, and every subsequent session depends on these types compiling correctly and these schemas validating real data. The output is a set of focused type files under `src/types/` that collectively describe all 16 entity types, their formalization subtypes, all value objects, all estimate wrappers, and the structural plumbing (`BaseEntity`, `CanonicalStore`, `EntityType`, `EntityRef`, `STORE_KEY`).

## Depends on

- Nothing. This is the first session.

## Behavior

- Rule: Every entity interface must include the optional `stale_markers?: ReadonlyArray<StaleMarker>` field from `BaseEntity`.
- Rule: An entity is considered stale if `stale_markers` is non-empty. Multiple causes can coexist.
- Rule: `mark_stale` appends to the array (skip if a marker with the same `caused_by` already exists).
- Rule: `clear_stale` removes the specific cause from the array. Entity becomes non-stale when the array empties.
- Rule: `readiness_cache` is a derived convenience field populated by the readiness engine (session 4.1). It is NOT serialized to `.gta.json`. On file load, it is `undefined` until recomputed. It must never be set directly — only via the readiness engine.
- Rule: `CrossGameLink.target_player_id` is REQUIRED (not just optional) when `effect_type` is one of: `payoff_shift`, `belief_update`, `strategy_unlock`, `strategy_remove`, `resource_transfer`. The Zod schema should validate this conditional requirement. Session 7.1 will reject links with missing `target_player_id` for these effect types with a clear error.
- Rule: `EntityType` has exactly 16 members. Adding or removing one is a schema-breaking change.
- Rule: `STORE_KEY` maps every `EntityType` to a pluralized key in `CanonicalStore`. The mapping is `game` -> `games`, `game_node` -> `nodes`, `game_edge` -> `edges`, `derivation` -> `derivations`. All others add `s`.
- Rule: `Formalization` is a discriminated union on the `kind` field with 7 subtypes. Each subtype extends `BaseFormalization`.
- Rule: `InformationSet.beliefs` when present must sum to 1.0 ± 0.001 and all keys must be members of `node_ids`.
- Rule: `EvolutionaryFormalization.initial_distribution` values must sum to 1.0 ± 0.001 and keys must match `strategy_types[*].id`.
- Rule: `EvolutionaryFormalization.fitness_matrix` must be complete: both dimensions must match `strategy_types[*].id` exactly.
- Rule: `BargainingFormalization.parties` must be a subset of the linked game's `players` list.
- Rule: `EstimateValue.confidence` is always 0-1. The Zod schema enforces `z.number().min(0).max(1)`.
- Rule: `Actor` is a tagged union on `kind`. Never use raw strings for actors.
- Rule: `emptyCanonicalStore()` returns a new object every call (no shared mutable state).
- Example: `refKey({ type: 'claim', id: 'claim_abc123' })` returns `"claim:claim_abc123"`.

## Guidance

- Organize types into focused files by domain (canonical, evidence, formalizations, estimates, auxiliary). Do not put everything in one file.
- Export Zod schemas alongside the TypeScript types. Use `z.infer<typeof schema>` to derive the TS type from the Zod schema, or define the interface first and use `satisfies` to verify alignment. Either approach is fine; pick one and be consistent.
- `CanonicalStore` uses ID-keyed `Record<string, T>` maps, not arrays. Arrays only appear in the `.gta.json` file format (handled in session 1.2). Do not create array-based store types.
- The `stale_markers` field is optional on every entity. Do not make it required — most entities are not stale.
- `ConditionRef` looks small but is used by `CrossGameLink`, `ForecastEstimate`, and the play-out engine. Define it carefully.
- Do not define `Command`, `ModelEvent`, `DispatchResult`, or any engine types here. Those belong to session 2.1.
- Do not define `AnalysisFile` here. That belongs to session 1.2.
- Watch out for circular imports between `estimates.ts` and `auxiliary.ts` — `ConditionRef` is used by `ForecastEstimate`, and `EstimateValue` is used by `CoalitionOption`. Plan the import graph before coding.
- The composition engine (session 7.1) needs to derive typed TargetField variants from CrossGameLink. Some TargetField variants require a player_id (e.g., terminal_payoff, available_strategies, belief_prior). The `target_player_id` field on CrossGameLink carries this information so the translation is deterministic without hidden inference.
- Rule: All function and type names must be self-documenting. Avoid abbreviations beyond standard ones (ref, id, config). A human reviewer should understand the purpose from the name alone. Examples of good names: `translateLinkToPatch`, `computeImpact`, `buildInverseIndex`. Examples of bad names: `proc`, `handle`, `doStuff`, `xform`.

> **C1 — Player `role` vs `type` (CRITICAL).** The existing `type` field (`'state' | 'organization' | 'individual' | 'coalition' | 'market' | 'public'`) describes *what kind of entity the player IS* — its organizational form.
