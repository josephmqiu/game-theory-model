# Session 1.3 ‚Äî "Schema migration works"

## Objective

Build the schema migration pipeline so that `.gta.json` files from older format versions can be loaded by newer app versions. Produce the `SchemaVersion` registry, `MigrationTransform` declarations, and the `migrateFile` function with per-step validation, lossy tracking, and recovery mode reporting. After this session, opening an old-format file triggers automatic migration with a clear audit of what changed and what was lost.

## Contracts

### Types to create

```typescript
interface SchemaVersion {
  format: number                             // 1, 2, 3, ...
  schema: ZodType<AnalysisFile>
  normalize: (data: AnalysisFile) => AnalysisFile
  released_at: string
  breaking: boolean
}

interface MigrationTransform {
  from: number
  to: number
  description: string
  lossy: boolean
  discarded_fields?: string[]
  transform: (data: unknown) => { result: unknown; discarded?: Record<string, unknown> }
}

type MigrationResult =
  | {
      status: 'success'
      data: AnalysisFile
      steps_applied: StepResult[]
      discarded_data?: Array<{
        step: { from: number; to: number }
        description: string
        fields: string[]
        values: Record<string, unknown>
      }>
    }
  | {
      status: 'migration_failed'
      failed_at_step: { from: number; to: number }
      description: string
      errors: unknown[]
      partial_data: unknown
    }
  | {
      status: 'unsupported_version'
      file_version: number
      app_version: number
      message: string
      recovery_available: true
    }

interface StepResult {
  from: number
  to: number
  status: 'ok'
}
```

## Behavior

### Migration path construction

- Rule: Every intermediate step must exist. `buildMigrationPath(1, 4)` requires transforms for 1->2, 2->3, and 3->4. If any is missing, throw `MigrationError` naming the gap.
- Rule: No skip-ahead transforms (e.g., 1->3 directly) are allowed.
- Rule: `buildMigrationPath(n, n)` returns an empty array (no migration needed).

### Migration execution

- Rule: For each step, run `transform(data)`, then validate the result against the target schema via `getSchema(step.to).schema.safeParse(data)`.
- Rule: If validation passes, run `normalize` to materialize all defaults.
- Rule: If validation fails at any step, return `migration_failed` with the step info, Zod errors, and the partial data at the point of failure.
- Rule: If a step is marked `lossy: true` and produces `discarded` data, collect it into the `discarded_data` array on the result.
- Example: Migrating v1 -> v3 runs transform 1->2 (validate against v2 schema, normalize), then transform 2->3 (validate against v3 schema, normalize). If step 2->3 discards a field, the final result includes that in `discarded_data`.

### Same-version loading

- Rule: When `from === to`, skip the migration pipeline but still validate and normalize through the current schema. This catches structurally invalid files even at the current version.

### Future version handling

- Rule: If `from > getCurrentSchemaVersion()`, return `unsupported_version` with a message suggesting the user update the app, and `recovery_available: true`.

### Normalization

- Rule: Each schema version's `normalize` function materializes all optional fields to their default values. After normalization, the saved file has every field present ‚Äî no relying on Zod `.optional()` defaults at runtime.
- Example: If a v2 schema adds an optional `quality_rating` field to `Source`, the v2 normalize function sets it to `'medium'` when absent.

### Integration with load pipeline

- Rule: The migration pipeline replaces the stub in step 2 of `loadAnalysisFile`. If migration returns `success`, step 3 (validate) uses the current schema. If migration returns `migration_failed` or `unsupported_version`, the load pipeline returns a recovery-mode result.

## Guidance

- Start with only format version 1 in the registry and zero real transforms. The infrastructure must work correctly with the degenerate case (current version, no migration needed). Real transforms are added as the schema evolves.
- Write one test transform (v1 -> v2) as a test fixture, not as production code. It should add a harmless optional field so you can verify the pipeline works end-to-end.
- **Known migration needed:** The `DiscountModel` type change on `RepeatedGameModel.discount_factors` converts `Record<string, EstimateValue>` to `Record<string, DiscountModel>`. The migration function wraps each existing `EstimateValue` as `{ type: 'exponential', delta: <existing_value> }`. This also changes `REFERENCE_SCHEMA` field paths from `discount_factors.*.source_claims` to `discount_factors.*.delta.source_claims`. Register this as a schema version increment when the type change lands.
- The `normalize` function is critical. Without it, two files at the same schema version can have different shapes (one has the optional field, one does not). Normalization eliminates that ambiguity. Every field must be present after normalization.
- `MigrationError` should carry structured data (which step failed, what was expected) not just a string message. Downstream code will need to render it.
- Do not store migration transforms inline in a giant object literal. Use a flat array of `MigrationTransform` objects that can be tested individually.
- The recovery mode UI is not built in this session. This session produces the data structures that recovery mode will display. Return the raw JSON and error details; let session 3.1 render them.
- Be careful with the `transform` function signature: it takes `unknown` and returns `unknown`. This is intentional ‚Äî each step operates on a different schema shape. Type safety comes from the per-step Zod validation, not from the transform's parameter type.
