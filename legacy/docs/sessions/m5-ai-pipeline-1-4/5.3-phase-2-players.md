# Session 5.3 — Phase 2: Player Identification

## Goal
AI identifies all players with types, objective functions, information asymmetries. Produces Player entity proposals.

## Depends on
- 5.2 (Phase 1 evidence in canonical store)
- 1.1 (Player type with new fields: priority_ordering, stability_indicator, non_standard_utility, information_state)

## Contract

### Phase 2 Output

```typescript
interface PlayerIdentificationResult {
  phase: 2
  status: PhaseResult
  proposed_players: ProposedPlayer[]
  information_asymmetry_map: InformationAsymmetryMap
  proposals: ModelProposal[]
}

interface ProposedPlayer {
  temp_id: string
  name: string
  type: 'state' | 'organization' | 'individual' | 'coalition' | 'market' | 'public'
  role: 'primary' | 'involuntary' | 'background' | 'internal' | 'gatekeeper'
  parent_player_id?: string // for 'internal' role — which primary player they're part of
  objectives: ProposedObjective[]
  priority_ordering: { objective_index: number; priority: 'absolute' | 'tradable' }[]
  stability_indicator: 'stable' | 'shifting' | 'unknown'
  non_standard_utility: string | null
  information_state: {
    knows: string[]
    doesnt_know: string[]
    beliefs: string[]
  }
  constraints: string[]
  evidence_refs: EntityRef[]
  confidence: EstimateValue
  rationale: string
}

interface ProposedObjective {
  description: string
  type: 'survival' | 'expansion' | 'reputation' | 'economic' | 'ideological' | 'other'
  internal_conflicts: string[] // conflicts with other objectives
}

interface InformationAsymmetryMap {
  entries: {
    player_a: string // temp_id
    player_b: string
    a_knows_about_b: string[]
    b_knows_about_a: string[]
    critical_gaps: string[]
  }[]
}
```

### Phase 2 Behavior
1. AI uses 'player_analysis' role
2. Input: all Phase 1 evidence from canonical store
3. Identifies players with entity type (State, Organization, Individual, Coalition, Market, Public) and role (Primary, Involuntary, Background, Internal, Gatekeeper)
4. For each player: writes explicit objective function with AND clauses, identifies internal conflicts, priority ordering
5. Flags non-standard utility (ideology, honor, historical trauma, regime survival logic)
6. Maps information asymmetries between all player pairs
7. Produces add_player command proposals via diff review
8. Conversation: "Identified 6 players. 3 proposals for your review." + inline proposal cards

### Revalidation Triggers
Phase 2 can trigger revalidation if:
- Internal player with independent agency discovered -> `trigger_revalidation({ trigger: 'new_player_discovered', ... })`
- This is logged as RevalidationEvent and shown in Phase 5 dashboard

## Behavior rules
- Player role 'internal' (intra-player agent) MUST have parent_player_id referencing the primary player
- Every player must have at least one objective with evidence support
- Information asymmetry map is advisory (not canonical entity) — helps Phase 3 build Bayesian games
