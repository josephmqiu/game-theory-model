# Session 5.4 — Phase 3: Baseline Model + Phase 4: Historical Repeated Game

## Goal
AI builds the baseline strategic model (Phase 3) and maps historical interactions (Phase 4). These are combined in one session because Phase 4 almost always triggers revalidation of Phase 3.

## Depends on
- 5.3 (Phase 2 players in canonical store)
- 1.1 (Game, Formalization, EscalationLadder, TrustAssessment, RepeatedGamePattern, DynamicInconsistencyRisk, CrossGameConstraintTable types)

## Contract

### Phase 3 Output

```typescript
interface BaselineModelResult {
  phase: 3
  status: PhaseResult
  proposed_games: ProposedBaselineGame[]
  escalation_ladder: ProposedEscalationLadder | null
  strategy_table: ProposedStrategyTable
  cross_game_constraint_table: ProposedCrossGameConstraintTable | null
  model_gaps: string[] // "What does this model fail to explain?"
  proposals: ModelProposal[]
}

interface ProposedBaselineGame {
  temp_id: string
  name: string
  canonical_type: CanonicalGameType
  players: string[] // player IDs
  description: string
  move_order: 'simultaneous' | 'sequential' // 'repeated' and 'incomplete_information' are game structure properties captured by canonical_type and formalization kind
  time_structure: {
    event_time: string // real-world timing
    model_time: string // strategic move order
    simulation_time: string // play-out turn order
  }
  deterrence_vs_compellence: 'deterrence' | 'compellence' | 'both' | 'neither' | null
  institutional_constraints: InstitutionalConstraint[]
  adjacent_game_test: 'changes_answer' | 'does_not_change_answer' | 'uncertain'
  evidence_refs: EntityRef[]
  confidence: EstimateValue
  rationale: string
}

interface InstitutionalConstraint {
  category: 'international_institution' | 'domestic_legal' | 'alliance_obligation' | 'economic_institution' | 'arms_control_framework' | 'regulatory' | 'other'
  description: string
  constraining_effect: string // how it constrains the game
  evidence_refs: EntityRef[]
}

type CanonicalGameType =
  | 'chicken_brinkmanship'
  | 'prisoners_dilemma'
  | 'coordination'
  | 'war_of_attrition'
  | 'bargaining'
  | 'signaling'
  | 'bayesian_incomplete_info'
  | 'coalition_alliance'
  | 'domestic_political'
  | 'economic_chokepoint'
  | 'bertrand_competition'
  | 'hotelling_differentiation'
  | 'entry_deterrence'
  | 'network_effects_platform'

interface ProposedEscalationLadder {
  game_id: string // temp_id of proposed game
  rungs: {
    label: string
    description: string
    reversible: boolean
    climbed: boolean
    player_attribution: string | null // player temp_id
    evidence_refs: EntityRef[]
    strategic_implications: string
  }[]
  escalation_dominance: string | null // player temp_id
  stability_instability_paradox: boolean
}

interface ProposedStrategyTable {
  players: string[] // player temp_ids (typically 2 primary players)
  strategies: {
    player_id: string
    strategies: {
      label: string
      feasibility: 'feasible' | 'rhetoric_only' | 'dominated' | 'uncertain'
      requirements: string[]
      evidence_refs: EntityRef[]
    }[]
  }[]
  // Outcome cells at strategy profile intersections (optional — ordinal only at Phase 3)
  outcome_cells: {
    strategy_profile: { player_id: string; strategy_label: string }[]
    outcome_description: string
    ordinal_rank_per_player: { player_id: string; rank: string }[] // e.g., "best", "worst", "2nd"
  }[] | null
}
```

### Phase 3 Behavior
1. AI uses 'baseline_modeling' role
2. Builds smallest game capturing main strategic tension (playbook 3a)
3. Matches to canonical game type from the 14-type list (playbook 3b)
4. Classifies deterrence vs compellence (playbook 3c)
5. Builds strategy table with feasibility annotations (playbook 3d)
6. Constructs escalation ladder if applicable (playbook 3e)
7. Specifies move order and time structure (playbook 3f)
8. Maps institutional constraints as game-structure parameters (playbook 3g)
9. Tests adjacent games: "Add only if it changes the answer" (playbook 3h)
10. Builds cross-game constraint table if multiple games (playbook 3i)
11. Explicitly answers: "What does this model fail to explain?"

### Phase 4 Output

```typescript

  game_still_correct: boolean
  revealed_repeated_not_oneshot: boolean

  hidden_type_uncertainty: boolean
  cooperative_equilibria_eliminated: boolean
  objective_function_changed: boolean

  revalidation_triggers: RevalidationTrigger[]
}
```

### Phase 4 Behavior
1. AI uses 'historical_analysis' role
2. Maps last 5-10 years of major interactions (playbook 4a)
3. Checks for 6 repeated-game patterns (playbook 4b): defection during cooperation, tit-for-tat, grim trigger, selective forgiveness, dual-track deception, adverse selection
4. Assesses trust infrastructure (playbook 4c): zero/low/moderate/high posterior belief
5. Identifies dynamic inconsistency risks (playbook 4d)
6. Documents global signaling effects (playbook 4e)
7. Re-checks baseline model with 6 questions (playbook 4f)
8. If baseline recheck triggers changes -> dispatches `trigger_revalidation` command

### Phase 3-4 Interaction
- Phase 3 runs first, produces baseline game proposals
- Phase 4 runs against Phase 3 output
- Phase 4's baseline recheck (4f) commonly triggers revalidation
- If revalidation fires: Phase 3 re-runs with Phase 4 findings, then Phase 4 re-checks
- Convergence typically in 2-3 passes

## Behavior rules
- Baseline game should be the SMALLEST game capturing main tension — avoid over-formalization
- Escalation ladder only if domain involves escalation dynamics
- Cross-game constraint table only if multiple games identified
- Phase 4 is the most common revalidation trigger point
- All proposals go through diff review in conversation
