# Session 5.2 — Phase 1: Situational Grounding

## Goal
AI researches the situation and produces structured evidence across the playbook's 7 categories.

### Pipeline Orchestrator and Shared Types

These types are used by ALL pipeline phases (M5-M7). Defined here because Phase 1 is the first phase to use them.

#### Analysis State

```typescript
interface AnalysisState {
  id: string
  event_description: string
  domain: string // from ClassificationResult
  current_phase: number | null // null = not started, 1-10 = active phase
  phase_states: Record<number, PhaseState>
  pass_number: number // which pass through the pipeline (1 = first, 2+ = after revalidation)
  status: 'not_started' | 'running' | 'paused' | 'complete' | 'failed'
  started_at: string | null
  completed_at: string | null
}

interface PhaseState {
  phase: number
  status: 'pending' | 'running' | 'complete' | 'needs_rerun' | 'skipped'
  pass_number: number // which pass this phase was last run on
  started_at: string | null
  completed_at: string | null
  phase_execution_id: string | null // links to PhaseExecution
}
```

Storage: L3 (SQLite). AnalysisState is operational pipeline state, not canonical model data.

#### Pipeline Orchestrator

```typescript
interface PipelineOrchestrator {
  // Start a new analysis from a description
  startAnalysis(description: string, options?: { manual?: boolean }): Promise<AnalysisState>

  // Run the next phase (or a specific phase)
  runPhase(phase: number): Promise<PhaseResult>

  // Pause the pipeline (user wants to inspect/edit before continuing)
  pause(): void

  // Resume after pause
  resume(): void

  // Cancel the current phase
  cancelCurrentPhase(): void

  // Get current state
  getState(): AnalysisState

  // Handle user steering message
  handleSteering(message: string): Promise<void>
}
```

The orchestrator:
1. Sequences phases 1-10, advancing only when the current phase completes
2. After each phase, checks revalidation triggers (via RevalidationEngine from session 6.1)
3. If revalidation fires, loops back to the earliest affected phase
4. Streams progress to the conversation UI (session 3.7) via structured messages
5. Pauses when proposals need user review (proposals appear inline in conversation)
6. Resumes when user has reviewed all pending proposals
7. Handles user steering messages by adjusting the current phase's behavior
8. Tracks all executions for cost/audit purposes

Storage: AnalysisState and PhaseExecution in SQLite (L3). Never in .gta.json.

### Phase 1 Behavior
1. AI uses 'research' role
2. Decomposes situation into research threads (one per evidence category, plus cross-cutting threads)
3. Threads run concurrently where provider supports it
4. Each finding produces Source + Observation + Claim proposals
5. "Actions vs Statements" category explicitly distinguishes what players have done from what they've said
6. All proposals go through diff review (user accepts/rejects in Overview conversation)
7. Progress streamed to conversation: "Researching capabilities... Found 8 data points..."
8. Phase completes when all categories have at least baseline coverage

### Conversation Integration
- Phase start message: "Starting Phase 1: Situational Grounding..."
- Progress updates: per-category findings as they arrive
- Completion message: "Phase 1 complete. [N] evidence items across [M] categories. [K] proposals for review."
- Proposals appear inline in conversation with accept/reject buttons
- User can type steering messages: "Focus more on economic impact" -> AI adjusts research priorities

## Behavior rules
- All AI outputs Zod-validated
- Malformed responses fail loudly with error in conversation
- No auto-retry; user decides
- Thread failures don't crash other threads
- Evidence normalization (Source -> Observation -> Claim derivation chains) happens within this phase, not as separate phase
