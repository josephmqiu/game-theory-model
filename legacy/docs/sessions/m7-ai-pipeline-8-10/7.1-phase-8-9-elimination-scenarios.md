# Session 7.1 — Phase 8: Elimination + Phase 9: Scenario Generation

## Goal
AI eliminates implausible outcomes (Phase 8), then generates scenario predictions with central thesis (Phase 9). Combined because elimination directly constrains scenario space.

## Depends on
- 6.3 (Phase 7 assumptions with sensitivity ratings)
- 6.2 (Phase 6 formalizations and equilibria)
- 1.1 (EliminatedOutcome, Scenario, CentralThesis types)

## Contract

### Phase 8 Output

```typescript
interface EliminationResult {
  phase: 8
  status: PhaseResult
  eliminated_outcomes: ProposedEliminatedOutcome[]
  proposals: ModelProposal[]
}

interface ProposedEliminatedOutcome {
  temp_id: string
  outcome_description: string
  elimination_reasoning: string
  citing_phases: { phase: number; finding: string }[]
  evidence_refs: EntityRef[]
  surprise_factor: 'high' | 'medium' | 'low'
  related_scenarios: EntityRef[]
}
```

### Phase 8 Behavior
1. AI uses 'elimination' role
2. For each plausible outcome, checks if any phase findings eliminate it
3. Each elimination must cite specific phase findings (e.g., "Phase 4 shows trust infrastructure is destroyed -> bilateral negotiated settlement eliminated")
4. Surprise factor: would a non-expert consider this outcome plausible? High surprise = most valuable elimination
5. The test from playbook: "If someone without this analysis would consider the outcome plausible, it is worth explicitly eliminating with reasoning"
6. Produces add_eliminated_outcome command proposals
7. Conversation: "4 outcomes eliminated. Most surprising: bilateral settlement -- here's why..."

### Phase 9 Output

```typescript
interface ScenarioGenerationResult {
  phase: 9
  status: PhaseResult
  proposed_scenarios: ProposedScenarioFull[]
  tail_risks: ProposedTailRisk[]
  central_thesis: ProposedCentralThesis
  probability_check: {
    sum: number // should be ~100%
    missing_probability: number // if not ~100%, how much is unaccounted
    warning: string | null
  }
  proposals: ModelProposal[]
}

interface ProposedScenarioFull {
  temp_id: string
  name: string
  narrative: {
    summary: string // 1-2 sentence overview
    causal_chain: { phase: string; event: string; leads_to: string }[] // temporal + causal progression
    full_text: string // complete narrative
  }
  probability: ForecastEstimate // matches canonical Scenario.estimated_probability
  key_assumptions: { assumption_ref: EntityRef; sensitivity: 'critical' | 'high' | 'medium' | 'low' }[]
  invalidation_conditions: string[]
  model_basis: EntityRef[] // which games drive this scenario
  cross_game_interactions: EntityRef[] // which cross-game links are essential
  forecast_basis: 'equilibrium' | 'discretionary' | 'mixed'
  forecast_basis_explanation: string // which parts are model-driven, which are judgment
}

interface ProposedTailRisk {
  temp_id: string
  event_description: string
  probability: ForecastEstimate // matches canonical Scenario.estimated_probability
  trigger: string
  why_unlikely: string
  consequences: string
  drift_toward: boolean // is current trajectory drifting toward this?
  drift_evidence: string | null
}

interface ProposedCentralThesis {
  statement: string
  falsification_condition: string
  evidence_refs: EntityRef[]
  assumption_refs: EntityRef[]
  scenario_refs: string[] // temp_ids of proposed scenarios
  forecast_basis: 'equilibrium' | 'discretionary' | 'mixed'
}
```

### Phase 9 Behavior
1. AI uses 'scenario_generation' role
2. Builds scenarios per playbook 9a: narrative, probability, assumptions, invalidation conditions, model basis, cross-game interactions
3. Separates baseline forecast from tail risks (playbook 9b)
4. Generates central thesis as falsifiable claim (playbook 9c)
5. Labels each scenario/thesis as equilibrium prediction vs discretionary forecast (playbook 9d)
6. Checks probability sum ~= 100%; warns if off
7. Tail risks get separate treatment -- low probability but high consequence
8. Each tail risk assessed for drift: "Is current trajectory moving toward this?"
9. Produces add_scenario, add_tail_risk, add_central_thesis, and related command proposals. Each `ProposedTailRisk` maps to an `add_tail_risk` command proposal (not inline data on a parent entity).
10. Uses correlated assumption clusters from Phase 7 -- scenarios should account for correlated changes, not treat assumptions as independent

### Conversation Integration
- Phase 8: "Examining 8 potential outcomes... 4 eliminated. [Inline eliminated outcome cards]"
- Phase 9: "Generating scenarios... Central thesis: '[thesis]'. 4 scenarios. [Inline scenario cards with probability bars]"
- Tail risks shown separately: "2 tail risks identified. [Inline tail risk cards]"
- User can adjust: "I think the probability of escalation is higher" -> AI re-calibrates

### Cross-Game Composition (Shared Infrastructure)
Phase 9 uses the composition engine contracts from old session 7.1 (EffectPatch, Resolver, etc.) when evaluating cross-game interactions within scenarios. These contracts are preserved from the old spec and shared with Phase 6i.

The CascadeAnalysisRun from old 7.2 is used to trace how effects propagate across games within each scenario.

## Behavior rules
- Eliminations cite specific phases -- no vague reasoning
- Scenarios must sum to ~100% -- warn if not
- Central thesis must be falsifiable -- includes specific falsification condition
- Equilibrium vs discretionary labels are prominent, not hidden badges
- Tail risks not buried because probability is low
- Correlated assumptions treated together, not independently
