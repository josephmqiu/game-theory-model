# Session 4.3 — "Backward induction + sensitivity"

## Objective

Implement backward induction for extensive-form games, Bayesian updating for incomplete information models, and the mandatory sensitivity analysis layer that applies to all solver results. Sensitivity is not optional — every computational result must answer what assumption changes the result, which payoff matters most, and what evidence would flip the conclusion. ## Behavior

### Backward induction

- Rule: Before running, call `checkSolverGate('backward_induction', ...)`. If not eligible, return `status: 'failed'`.
- Rule: Walk the tree from terminal nodes upward. At each decision node, the acting player picks the edge maximizing their payoff.
- Rule: At chance nodes, compute expected payoff weighted by `chance_estimate.value` on outgoing edges.
- Rule: `solution_path` traces the equilibrium path from root to the terminal node reached under optimal play.
- Rule: `subgame_values` records the computed value at every non-terminal node, including the alternatives not chosen and the payoff difference.
- Rule: If any terminal node lacks payoffs, return `status: 'partial'` with a warning listing the nodes without payoffs.
- Rule: Information sets affect which nodes a player can distinguish. If two nodes are in the same information set, the player must choose the same action at both.

### Information set handling algorithm

For backward induction with information sets:

1. Build the game tree from `root_node_id`, traversing edges depth-first.
2. At each decision node, check if it belongs to an `InformationSet` (via `information_set_id`).
3. If the node is NOT in an information set, compute the optimal action normally: for each outgoing edge, compute the subtree value, pick the action that maximizes the acting player's payoff.
4. If the node IS in an information set:
   a. Collect all nodes in the same information set.
   b. For each available action A (shared across all nodes in the set):
      - Compute the expected payoff of action A at each node in the set.
      - If `InformationSet.beliefs` is provided, weight payoffs by those probability weights.
      - If `InformationSet.beliefs` is absent, use uniform weights (equal probability per node).
   c. The player picks the action that maximizes the weighted average payoff across all nodes in the set.
   d. This action is assigned to ALL nodes in the set (consistent play).
   e. When uniform beliefs are used, the `BackwardInductionResult.meta` must include: `limitations: ["Assumes uniform beliefs at information sets — not equivalent to sequential equilibrium"]`.
5. If the information set has nodes at different depths, this indicates an invalid formalization. Return `status: 'failed'` with error "Information set spans different tree depths."

Rule: Backward induction assumes perfect recall — each player remembers their own past moves. If a formalization violates perfect recall (a player's information set contains nodes where the player took different past actions), add a readiness warning: "Perfect recall may be violated at information set X."

### Bayesian updating

- Rule: Before running, call `checkSolverGate('bayesian_update', ...)`.
- Rule: Start from prior distributions over player types.
- Rule: For each observation (signal), apply Bayes' rule: `posterior(type) = prior(type) * likelihood(observation|type) / normalizer`.
- Rule: Track the full update chain so each step is inspectable.
- Rule: If a likelihood is zero for all types, return `status: 'failed'` with "Observation inconsistent with all types."
- Rule: Label the result as "Approximate Bayesian update (heuristic)" when inputs are estimated rather than precise.

### Likelihood source for Bayesian updates

- Rule: Likelihood values `P(observation | type)` come from `BayesianGameModel.signal_structure.signals`. Each signal entry maps `{ label, type_label, probability }` where `probability` is the likelihood of observing that signal given that player type.
- Rule: If `signal_structure` is absent or empty, Bayesian update cannot proceed. Return `status: 'failed'` with error "Signal structure required for Bayesian updating."
- Rule: If a type has no matching signal entry, use likelihood = 0 (type is ruled out by the observation).
- Rule: If all likelihoods are zero for all types, return `status: 'failed'` with error "Observation rules out all types — check signal structure."
- Rule: If any computed posterior is < 1e-10, clamp to 1e-10 and add warning "Near-zero posterior for type X — numerical precision limit reached."

### Sensitivity analysis (mandatory)

- Rule: Every solver result must be accompanied by sensitivity analysis. This is not optional.
- Rule: `analyzeSensitivity` takes a completed solver result and systematically varies each input.
- Rule: For each payoff in the formalization, compute the threshold value at which the solver's conclusion changes (e.g., equilibrium shifts, backward induction path changes).
- Rule: `payoff_sensitivities` are sorted by margin (smallest margin = most sensitive = reported first).
- Rule: For each assumption referenced by the formalization, determine whether negating it would change the result.
- Rule: `overall_robustness` is `'robust'` if the smallest margin exceeds 20% of the payoff range, `'sensitive'` if 5-20%, `'fragile'` if below 5%.
- Rule: Sensitivity results answer these four questions explicitly:
  1. What assumption changes this result?
  2. Which payoff estimate matters most?
  3. Which latent factor is load-bearing?
  4. What evidence update would flip the conclusion?

### Tree view integration

- Rule: The backward induction solution path is highlighted in the tree view.
- Rule: Non-optimal edges are dimmed but remain visible with their payoff information.
- Rule: Each node shows its subgame value on hover.
- Rule: Sensitivity information appears in a side panel: "Changing payoff X by Y flips the result."

## Guidance

- Backward induction is a recursive algorithm. Implement it as a depth-first traversal from leaves to root. Do not try to flatten it into iteration unless the tree is very deep.
- Information sets are the hard part of backward induction. When two nodes share an information set, the player must choose the same action at both. Handle this by grouping nodes by information set before computing optimal actions.
- Sensitivity analysis should be implemented as a generic layer that works with any `SolverResult`, not hardcoded per solver. The solver-specific logic is only "what does it mean for the result to change" — for Nash, it means a different equilibrium; for backward induction, it means a different path.
- Do not re-run the full solver for every payoff perturbation. For small games, perturbation is fine. For larger games, use analytical sensitivity (derivative of the solution with respect to each parameter) where possible.
- The four sensitivity questions are user-facing. The `SensitivityAnalysis` type must have fields that directly answer them. Do not bury the answers in nested structures.
- Robustness thresholds (5%, 20%) are heuristics, not theorems. Document them as configurable defaults.
- Bayesian updating is simpler than it sounds. The hard part is structuring the inputs (priors, likelihoods) correctly from the `BayesianGameModel`. The math is just Bayes' rule repeated.
