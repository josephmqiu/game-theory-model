# Session 4.1 — "Solver readiness gates work"

## Objective

Build the solver readiness engine that evaluates whether a formalization has sufficient structure, data, and confidence for each solver to produce meaningful results. This is the gatekeeper that prevents the app from showing polished-looking results built on incomplete or low-confidence inputs. Every solver in M4 depends on readiness checks, so this session must land first.

## Contracts

### Types to create

```typescript
interface SolverReadiness {
  overall: 'ready' | 'usable_with_warnings' | 'not_ready'
  completeness_score: number        // 0-1
  confidence_floor: number          // Lowest confidence among key inputs
  blockers: string[]
  warnings: string[]
  supported_solvers: SolverKind[]
}

type SolverKind =
  | 'nash'
  | 'backward_induction'
  | 'expected_utility'
  | 'dominance'
  | 'bayesian_update'
  | 'cascade'
  | 'simulation'
  // Deferred solvers (DEFERRED:M9/M10) — gates return not_ready:
  | 'bargaining'
  | 'evolutionary'
  | 'correlated_equilibrium'
  | 'credible_commitment'
  | 'game_classifier'
  | 'mechanism_design'

interface SolverRequirement {
  solver: SolverKind
  check: (formalization: Formalization, store: CanonicalStore) => SolverGateResult
}

interface SolverGateResult {
  eligible: boolean
  blockers: string[]
  warnings: string[]
  completeness: number              // 0-1, fraction of requirements met
  confidence_floor: number          // lowest confidence among inputs used
}

interface ReadinessReport {
  formalization_id: string
  computed_at: string
  readiness: SolverReadiness
  per_solver: Record<SolverKind, SolverGateResult>
}
```

## Behavior

### Solver coverage in M4

- Normal-form: Nash equilibrium, dominance elimination, expected utility (session 4.2)
- Extensive-form: Backward induction (session 4.3)
- Bayesian: Belief updating only — no equilibrium solver (session 4.3)
- Repeated: NOT COVERED in M4. Readiness gate requires `equilibrium_selection` field (not just generic "no solver") — returns `not_ready` with blocker "Repeated game solver requires equilibrium_selection criterion (grim_trigger, tit_for_tat, etc.)."
- Coalition: NOT COVERED in M4. Readiness gate requires `solution_concept` and `characteristic_function` — returns `not_ready` with blocker "Coalition solver requires solution_concept and characteristic_function."
- Bargaining: DEFERRED:M9. Returns `not_ready` with blocker "Bargaining solver not yet implemented."
- Evolutionary: DEFERRED:M9. Returns `not_ready` with blocker "Evolutionary solver not yet implemented." Evolutionary formalizations are exempt from the "at least 2 defined players" check (they use anonymous strategy types, not named players).
- Correlated equilibrium, credible commitment, game classifier: DEFERRED:M9. Return `not_ready`.
- Mechanism design: DEFERRED:M10. Returns `not_ready`.

Rule: For formalization kinds without solvers, `computeReadiness` must return `overall: 'not_ready'` with a specific blocker message describing what's needed (not just "no solver available"). The user can still model these games but cannot run computation on them.

### Readiness gate rules per solver

| Solver | Minimum requirements |
|--------|----------------------|
| Nash | Complete strategy set for all players, usable payoff representation (cardinal or interval), exactly 2 defined players (n>2 is PPAD-complete, DEFERRED:M9). If any payoff uses `ordinal_rank`, mixed-strategy Nash is hard-blocked with "Mixed-strategy Nash requires cardinal payoffs." Pure-strategy Nash and dominance remain available on ordinal. |
| Backward induction | Ordered tree with root node, terminal payoffs on all leaf nodes, information sets defined |
| Expected utility | Probabilities or conditional forecasts on chance/choice edges, terminal payoffs present |
| Bayesian update | Prior distributions defined, player types specified, observation-to-belief mapping exists |
| Cascade | At least one typed cross-game link with valid source and target references |
| Dominance | Complete strategy set, usable payoff representation, at least 2 players |
| Simulation | At least one formalization with nodes and edges, at least one player assigned |

### Overall readiness computation

- Rule: `overall` is `'ready'` when at least one solver has `eligible: true` and zero blockers exist.
- Rule: `overall` is `'usable_with_warnings'` when at least one solver is eligible but warnings exist (stale inputs, low confidence, wide estimate ranges).
- Rule: `overall` is `'not_ready'` when no solver is eligible.
- Rule: `completeness_score` is the max completeness across all per-solver checks.
- Rule: `confidence_floor` is the minimum confidence value found across all `EstimateValue` fields that the eligible solvers would consume.
- Rule: If any assumption referenced by the formalization has non-empty `stale_markers`, add a warning: "Assumption '{statement}' is stale: {reason}" (using the first marker's reason).
- Rule: If any payoff uses `representation: 'ordinal_rank'` and the solver requires cardinal values, add a blocker.

### Warning generation

- Rule: If payoffs use `representation: 'interval_estimate'` with `(max - min) > 0.5 * max`, add warning: "Wide interval estimates may produce unreliable equilibria."
- Rule: If confidence on any key input is below 0.3, add warning: "Low confidence input ({field}): {confidence}."
- Rule: If more than half the assumptions referenced by a formalization lack `supported_by` evidence, add warning: "Majority of assumptions lack evidence support."
- Rule: Warnings are informational. They never prevent a solver from running — only blockers do that.

### Readiness caching and invalidation

- Rule: Readiness reports are cached in the Layer 3 derived store, keyed by `formalization_id`.
- Rule: The cache is invalidated when any command modifies: the formalization itself, any entity referenced by the formalization (players, nodes, edges, assumptions), or any `EstimateValue` contained within those entities.
- Rule: Invalidation triggers lazy recomputation — the next read of the report recomputes it. Do not eagerly recompute on every command.
- Rule: The `computed_at` timestamp on the report lets the UI show whether the readiness is fresh or stale relative to the last model edit.

### Edge cases

- A formalization with zero players produces `not_ready` with blocker "No players assigned" — except for `kind: 'evolutionary'` which uses anonymous strategy types instead of named players.
- A Nash gate on a game with more than 2 players produces `not_ready` with suggestions: "Consider dominance elimination", "Consider pairwise 2-player subgame analysis", "Use play-out engine for interactive exploration."
- A normal-form formalization with empty `strategies` map produces blocker "No strategies defined for any player".
- A formalization whose `readiness` field is stale triggers recomputation.
- Readiness is recomputed whenever a command modifies the formalization or its referenced entities.
- A repeated game formalization whose `stage_formalization_id` points to a non-existent formalization produces blocker "Stage game not found."
- A coalition formalization with zero `coalition_options` produces blocker "No coalition options defined."

## Guidance

- Readiness is a pure computation over the canonical store. It has no side effects and no state of its own beyond the cached report.
- Do not conflate readiness with solver results. Readiness says "you can run this solver." It does not run the solver.
- The per-solver gate functions should be independently testable. Register them in a lookup table keyed by `SolverKind`.
- Stale markers on referenced entities should produce warnings, not blockers. Stale never blocks editing or computation; it warns.
- Do not hardcode the readiness check into individual solvers. Solvers should call `checkSolverGate` before running, but the gate logic lives in `readiness.ts`.
- Confidence floor should walk all `EstimateValue` objects reachable from the formalization's nodes, edges, and payoffs. Do not traverse the entire store.
- The ReadinessPanel should display per-solver status as a list: solver name, eligible/not, and the specific blockers or warnings. Not a single pass/fail badge.
