# Session 4.2 — "Nash + dominance + expected utility"

## Objective

Implement the three core solvers for normal-form games: Nash equilibrium computation, dominance elimination, and expected utility calculation. Results are stored in Layer 3 (derived), displayed in the matrix view with highlighted equilibria and dominated strategies, and always accompanied by readiness metadata and solver-specific warnings. This session delivers the first computational output users can see.

## Contracts

### Types to create

```typescript
interface SolverResultMeta {
  method_id: string           // e.g., 'nash_support_enumeration'. Convention: {solver_kind}_{variant}
  method_label: string        // Human-readable, e.g., 'Nash Equilibrium (Support Enumeration, 2-Player)'
  limitations: string[]       // Structural limitations of the method (inherent to the algorithm)
  assumptions_used: string[]  // Implicit assumptions the method makes
}

interface SolverResult {
  id: string
  formalization_id: string
  solver: SolverKind
  computed_at: string
  readiness_snapshot: SolverReadiness
  status: 'success' | 'partial' | 'failed'
  warnings: string[]                      // Runtime warnings (vary per invocation)
  meta: SolverResultMeta                  // Structural metadata (inherent to the algorithm)
  sensitivity?: SensitivitySummary
}

interface NashResult extends SolverResult {
  solver: 'nash'
  equilibria: NashEquilibrium[]
  method: 'support_enumeration' | 'lemke_howson' | 'vertex_enumeration'
}

interface NashEquilibrium {
  id: string
  strategies: Record<string, MixedStrategy>
  payoffs: Record<string, number>
  type: 'pure' | 'mixed'
  stability: 'strict' | 'weak'
}

interface MixedStrategy {
  player_id: string
  distribution: Record<string, number>  // strategy_label -> probability
}

interface DominanceResult extends SolverResult {
  solver: 'dominance'
  eliminated_strategies: DominatedStrategy[]
  reduced_game: {
    strategies: Record<string, string[]>
    remaining_cells: NormalFormCell[]
  }
  rounds: DominanceRound[]
}

interface DominatedStrategy {
  player_id: string
  strategy: string
  dominated_by: string | string[]     // pure or mixed
  dominance_type: 'strict' | 'weak'
  round: number
}

interface DominanceRound {
  round: number
  eliminated: DominatedStrategy[]
}

interface ExpectedUtilityResult extends SolverResult {
  solver: 'expected_utility'
  player_utilities: Record<string, PlayerUtility>
}

interface PlayerUtility {
  player_id: string
  per_strategy: Record<string, UtilityEstimate>
  best_response: string
}

interface UtilityEstimate {
  expected_value: number
  min_value?: number
  max_value?: number
  confidence: number
  assumptions_used: string[]
}

interface SensitivitySummary {
  most_sensitive_payoff: {
    player_id: string
    strategy_profile: string[]
    sensitivity_magnitude: number
  }
  result_change_threshold: number
  assumption_sensitivity: Array<{
    assumption_id: string
    impact: 'result_changes' | 'result_stable'
    description: string
  }>
}
```

## Behavior

### Nash equilibrium

- Rule: Before running, call `checkSolverGate('nash', ...)`. If not eligible, return a `SolverResult` with `status: 'failed'` and the gate's blockers as warnings.
- Rule: Start with 2-player normal-form games. Support enumeration of pure strategy Nash equilibria first.
- Rule: For 2x2 games, also compute mixed strategy equilibria analytically.
- Rule: Each equilibrium reports its type (`pure` or `mixed`) and stability (`strict` or `weak`).
- Rule: If payoffs use `interval_estimate`, compute equilibria using midpoint values and add a warning: "Equilibria computed on midpoint of interval estimates."
- Rule: Every `NashResult` carries a `readiness_snapshot` capturing the readiness state at solve time.

### Dominance elimination

- Rule: Perform iterated elimination of strictly dominated strategies first, then optionally weak dominance.
- Rule: Track elimination in rounds. Round 1 finds directly dominated strategies; round 2+ finds strategies dominated after prior eliminations.
- Rule: `reduced_game` contains the surviving strategy profiles and payoff cells.
- Rule: If no strategies are dominated, return an empty `eliminated_strategies` array with a note.

### Expected utility

- Rule: For each player, compute expected utility of each pure strategy given a probability distribution over opponent strategies.
- Rule: If no probability distribution is specified, use uniform distribution and add warning: "Using uniform distribution over opponent strategies."
- Rule: Report `best_response` as the strategy with highest expected utility.
- Rule: When payoffs are interval estimates, report min/max utility range.

### Results storage

- Rule: All results are stored in Layer 3 (derived). They never modify Layer 1.
- Rule: Results are keyed by `formalization_id` + `solver` + `computed_at`.
- Rule: Rerunning a solver replaces the previous result for that formalization+solver pair.

### Matrix view integration

- Rule: Nash equilibria cells are visually highlighted in the matrix view.
- Rule: Dominated strategies are shown with strikethrough or dimmed styling.
- Rule: Hovering an equilibrium cell shows the equilibrium details (type, stability, payoffs).
- Rule: The readiness summary is displayed above the solver results panel.

## Guidance

- Start with 2-player games only. N-player Nash is a separate concern for later optimization.
- Support enumeration is the simplest algorithm for small games. Do not reach for Lemke-Howson until 2-player works end-to-end.
- Solver functions should be pure: take formalization + store, return result. No side effects, no store mutations.
- Always extract the numeric value from `EstimateValue` explicitly. Never treat `EstimateValue` as a bare number.
- The matrix view highlight logic should be driven by the solver result data, not by solver internals. The view reads results from the derived store.
- Do not build sensitivity analysis into the solvers themselves. Session 4.3 handles that. But do include the `SensitivitySummary` type in the result so 4.3 can populate it later.
- Watch for division by zero in mixed strategy computation when a player has identical payoffs across strategies.
