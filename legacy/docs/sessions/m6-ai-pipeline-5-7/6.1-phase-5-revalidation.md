# Session 6.1 — Phase 5: Recursive Revalidation Mechanism

## Goal
Implement the revalidation mechanism as a persistent system that can fire from any phase, not just "Phase 5." Phase 5 in the sidebar is a dashboard/log of revalidation events.

Also: prompt versioning as a cross-cutting concern for all phases.

## Depends on
- 2.1 (command spine: trigger_revalidation command)
- 2.3 (stale persistence: revalidation events in SQLite)
- 5.1 (MCP server — ToolContext for re-running phases via command spine)
- 3.7 (conversation UI for revalidation messages)

## Preserves from old 6.3
- PromptVersion, PromptRegistry, PromptFork types
- Prompt management functions (getActivePrompt, forkPrompt, compareRuns)
- Stored in SQLite, not .gta.json

## Contract

### Revalidation Engine

```typescript
interface RevalidationEngine {
  // Check if any trigger conditions are met after a phase completes
  checkTriggers(phaseResult: PhaseResult, phase: number): RevalidationCheck

  // Execute revalidation: mark stale, re-run affected phases
  executeRevalidation(event: RevalidationEvent): Promise<RevalidationOutcome>

  // Get revalidation history for dashboard
  getRevalidationLog(analysisId: string): RevalidationEvent[]

  // Get current pass number (how many times we've looped)
  getCurrentPass(): number
}

interface RevalidationCheck {
  triggers_found: RevalidationTrigger[]
  affected_phases: number[]
  affected_entities: EntityRef[]
  recommendation: 'revalidate' | 'monitor' | 'none'
  description: string
}

interface RevalidationOutcome {
  event_id: string
  phases_rerun: number[]
  entities_marked_stale: EntityRef[]
  entities_updated: EntityRef[]
  new_pass_number: number
  converged: boolean // true if re-run produced no further triggers
}
```

### 11 Trigger Conditions (from playbook)

| # | Trigger | Loops back to | Stale propagation |
|---|---------|---------------|-------------------|
| 1 | New player with independent agency | Phase 2 | All games involving related entities |
| 2 | Player's objective function changed | Phase 2 + 3 | Games, formalizations, scenarios |
| 3 | New game identified | Phase 3 | Cross-game links, constraint tables |
| 4 | Existing game reframed | Phase 3 | Formalizations, equilibria, scenarios |
| 5 | Repeated interaction dominates one-shot | Phase 3 + 4 | Baseline model, formalizations |
| 6 | New cross-game link | Phase 3i or 6i | Constraint tables, scenarios |
| 7 | Escalation ladder needs revision | Phase 3e | Strategy tables, scenarios |
| 8 | Institutional constraint changed | Phase 3g | Strategy spaces, formalizations |
| 9 | Critical empirical assumption invalidated | Phase 7 | Scenarios, eliminated outcomes, equilibrium results |
| 10 | Model cannot explain core fact | Phase 3 | Everything downstream |
| 11 | Behavioral overlay changes prediction | Phase 6h | Scenarios, forecasts |

### Revalidation Behavior
1. After each phase completes, RevalidationEngine.checkTriggers() runs
2. If triggers found and recommendation is 'revalidate':
   a. Create RevalidationEvent entity via add_revalidation_event command
   b. Mark affected entities stale via trigger_revalidation command
   c. Show revalidation message in conversation: "Phase 4 broke the baseline model. Looping back to Phase 2."
   d. Update phase progress bar: warning on affected phases
   e. Re-run affected phases starting from earliest
3. After re-run, check triggers again
4. Convergence: typically 2-3 passes. After 4+ passes with structural changes still occurring, warn the user: "Analysis has not converged after N passes. This may indicate genuinely complex situation, inadequate information, or a wrong abstraction." User decides whether to continue or stop. No automatic hard stop.
5. User can dismiss a trigger (choose not to revalidate) — logged in event with resolution: 'dismissed'

### Phase 5 Dashboard (UI)
- Not where revalidation "happens" — it's a view/log
- Shows all RevalidationEvent entities, ordered by timestamp
- Each event shows: trigger condition, source phase, target phases, resolution, pass number
- Links to affected entities (clickable -> inspector)
- Convergence indicator: "Analysis converged after 2 passes" or "Still iterating — pass 3"

### Prompt Versioning (Cross-Cutting)

```typescript
interface PromptVersion {
  id: string
  phase: number
  name: string
  content: string
  parent_id: string | null
  is_official: boolean
  created_at: string
  metadata: { author: 'system' | 'user'; description: string; tags: string[] }
}

interface PromptRegistry {
  versions: Record<string, PromptVersion>
  active_versions: Record<number, string> // phase -> version_id
  official_versions: Record<number, string>
}
```

Each pipeline phase references a prompt version. Users can fork and customize prompts per phase. Stored in SQLite.

## Behavior rules
- Revalidation is not automatic — user sees the trigger and approves/dismisses
- But stale marking IS automatic (entities are immediately marked stale when trigger detected)
- Re-running a phase uses the same diff review mechanism (proposals in conversation)
- Pass number increments globally, not per-phase
- After 4+ passes with structural changes still occurring, the system warns the user: "Analysis has not converged after N passes. This may indicate genuinely complex situation, inadequate information, or a wrong abstraction." User decides whether to continue or stop. No automatic hard stop.

## Implementation Deviations

1. **Prompt registry storage**: Spec says SQLite. Implementation uses Zustand + browser
   persistence adapter (localStorage). Rationale: no SQLite infrastructure exists in the
   app; localStorage provides equivalent persistence for a local-first desktop app.
   The adapter can be swapped for SQLite later without API changes.

2. **RevalidationEngine.checkTriggers signature**: Spec takes `PhaseResult`;
   implementation takes `unknown` (the full phase output object, not just PhaseResult).
   Rationale: trigger detection needs the full phase output (e.g., proposed_players,
   baseline_recheck), not just the status summary.

3. **Trigger #1 (new_player_discovered) routes to [3,4] not [2]**: Phase 2 just
   completed when this fires, so re-running Phase 2 is redundant. Downstream phases
   (3, 4) need to incorporate the new player.
