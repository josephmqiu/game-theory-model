# Session 6.2 — Phase 6: Full Formal Modeling

## Goal
AI applies game-theoretic tools to the stabilized strategic picture. This is the most complex phase, with 9 sub-sections from the playbook (6a-6i). Uses M4 solvers.

## Depends on
- 5.4 (Phase 3-4 baseline model and history in canonical store)
- 6.1 (revalidation mechanism — Phase 6 can trigger revalidation)
- 4.1-4.3 (solvers for equilibrium computation, sensitivity analysis)
- 1.1 (Formalization types, SignalClassification)

## Contract

### Sub-section Contracts

**6a: Formal Representations**
```typescript
interface FormalRepresentationResult {
  proposed_formalizations: ProposedFormalization[]
  // One game can have multiple formalizations answering different questions
  complementary_note: string | null // explains why multiple are needed
}

interface ProposedFormalization {
  game_ref: EntityRef
  kind: 'normal_form' | 'extensive_form' | 'repeated' | 'bayesian' | 'signaling' | 'bargaining'
  purpose: string // what question this formalization answers
  abstraction_level: 'minimal' | 'moderate' | 'detailed'

  assumptions_introduced: ProposedAssumption[]
  evidence_refs: EntityRef[]

  rationale: string
}
```

**6b: Payoff Estimation**

### Phase 6 Behavior
1. AI uses 'formalization' role
2. Runs sub-sections sequentially (6a -> 6b -> ... -> 6i) since each builds on prior
3. Uses M4 solvers for 6c equilibrium computation
4. 6h behavioral overlays MUST be labeled "ADJACENT — NOT CORE GAME THEORY" in all output
5. 6i cross-game effects only computed if multiple games exist
6. Each sub-section produces proposals via diff review
7. Conversation shows sub-section progress: "6a: Choosing formal representations... 6b: Estimating payoffs..."
8. Phase 6 can trigger revalidation (e.g., formal modeling reveals hidden game -> trigger 'new_game_identified')

## Behavior rules
- Ordinal before cardinal for payoffs (6b) — never fabricate numerical precision
- Simple before fancy for equilibria (6c) — baseline equilibrium is anchor
- Behavioral overlays explicitly labeled so user knows which predictions are model-driven vs judgment
- All four adjacent tools (prospect theory, biases, scenario planning, red-teaming) must be explicitly acknowledged and labeled. If scenario planning was used to inform scenarios, note it. If red-teaming informed the adversarial challenge, note it. The point is transparent labeling of methodology.
- Cross-game effects use the composition engine from M7.1 (shared infrastructure)
- All sub-sections produce structured proposals, not prose

## Implementation Deviations

1. **Flat result structure vs nested sub_results**: Spec uses
   `FormalizationResult.sub_results.representations`. Implementation uses flat fields
   (`formal_representations`, `payoff_estimation`, etc.) plus `subsection_statuses`,
   `proposal_groups`, and `workspace_previews`. Rationale: flat fields are simpler to
   destructure and the additional fields (subsection_statuses, workspace_previews) don't
   fit the spec's `sub_results` nesting.

2. **Sub-result types are summary-oriented**: Spec contracts define per-entity detail
   types (e.g., `ProposedFormalization` with full structure). Implementation uses summary
   types (e.g., `FormalizationRepresentationSummary`) because full detail is in the
   canonical store after proposal acceptance. Phase results are status snapshots, not
   the canonical data itself.

3. **PayoffEstimationResult**: Spec has `estimates: PayoffEstimate[]` with
   `ordinal_rank: string`. Implementation has `updates: PayoffEstimationSummary[]`
   with `ordinal_first: boolean`. Ordinal rankings are stored on the formalization
   entities themselves (canonical layer), not duplicated in the phase result.
