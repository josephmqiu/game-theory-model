## Behavior

### Seven Benchmark Categories

1. **Textbook games** — PD, Chicken, coordination, signaling toy model. Purpose: solver correctness and UI sanity. The system should identify the correct game structure, compute correct equilibria, and display them in all relevant views.

2. **Historical crisis** — the founding US-Iran 2026 case (or equivalent multi-game scenario). Purpose: multi-game identification, linked-game reasoning, uncertainty handling. The system must identify multiple interacting games, create cross-game links, and handle deep uncertainty.

3. **Historical sequential crisis** — e.g., Cuban Missile Crisis or similar sequential deterrence case. Purpose: tree modeling, backward induction, signaling. The system must build extensive-form trees and compute backward induction correctly.

4. **Business / regulatory strategy** — antitrust, standards war, cartel coordination, platform bargaining. Purpose: non-military generality. The system must handle business-domain strategic analysis.

5. **Labor / negotiation** — strike, repeated bargaining, deadline pressure. Purpose: attrition, repeated interaction, audience-cost dynamics. The system must model repeated games and time-dependent payoffs.

6. **Live update benchmark** — a case with evidence arriving in stages. Purpose: update discipline, stale-model detection, scenario revision. The system must correctly mark stale entities and revise scenarios when evidence changes.

7. **Counterfactual benchmark** — intentionally perturb one major assumption. Purpose: sensitivity and invalidation quality. The system must show how a single assumption change cascades through the model.

### Eight Benchmark Metrics

- **Provenance score**: Can claims and estimates be traced cleanly to sources? Every estimate must have `source_claims`, every claim must have `based_on`, every inference must have `derived_from`.
- **Epistemic layering score**: Are fact, claim, inference, and assumption properly separated? No claim should be treated as an inference; no assumption should masquerade as a fact.
- **Formal adequacy score**: Did the system use the right solver on the right structure? Nash should not run on incomplete games; backward induction should not run without ordered trees.
- **Uncertainty calibration**: Are ranges and confidence levels sensible? No bare numbers. Wide ranges should have lower confidence. Critical assumptions should have sensitivity analysis.
- **Prediction coherence**: Do scenarios update coherently when evidence changes? Stale scenarios should show updated invalidators. New evidence should propagate to affected predictions.
- **Playout realism**: Are simulation branches internally consistent and well-rationalized? Every turn must have rationale. Cross-game effects must propagate. No contradictory moves within a branch.
- **Bundle reproducibility**: Can the analysis be replayed from the bundle? Export -> import -> verify checksums -> model matches.
- **Coverage**: Did the model find the relevant games, players, and dependencies? Measured against the fixture's `expected_games` and `expected_players`.

### Cost / Quota Tracking

- Rule: Every AI provider call is logged with provider, model, phase, tokens, and cost.
- Rule: The cost dashboard shows totals by provider, phase, and model.
- Rule: Users can set a quota with a warning threshold. When remaining cost drops below the threshold, a warning is shown.
- Rule: Cost records are persisted to SQLite.

### Provider Polish

- Rule: OpenRouter provider connects via the OpenRouter API with model selection and cost tracking.
- Rule: Local/Ollama provider connects to any OpenAI-compatible endpoint for offline or private use.
- Rule: Local providers cannot perform live web research. The system warns when selecting a local model for research phases.
- Rule: Capability detection identifies: web search availability, tool use, max concurrency, context window, streaming support, structured JSON reliability.

### Preloaded Examples

- Rule: The app ships with preloaded example analyses that demonstrate key capabilities.
- Rule: Examples load as complete `AnalysisFile` objects with all entity types populated.
- Rule: Examples pass provenance checks -- every estimate traces to claims, every claim traces to sources.

## Guidance

- Benchmark fixtures should be small, focused `.gta.json` files. Do not try to create exhaustive real-world analyses as fixtures -- use simplified models that test specific capabilities.
- The textbook benchmark is the minimum viable acceptance test. Start there and expand to other categories incrementally.
- Metrics should have clear thresholds. A score of 0.8+ should pass for most metrics; bundle reproducibility should require 1.0 (exact match).
- Cost tracking is append-only. Never delete cost records. The dashboard aggregates from the full history.
- OpenRouter and local providers should implement the same provider interface as the existing providers from session 5.1. Do not create a separate abstraction.
- Preloaded examples serve as both onboarding material and regression tests. If an example breaks, it should fail a test.
- Do not hardcode API keys or provider URLs. All provider configuration comes from user settings.
- The benchmark harness should be runnable from both the UI (a benchmark panel) and the command line (for CI). Design the harness as a pure function that the UI wraps.
- This is the final session. Prioritize making the existing system robust over adding new features. Fix any rough edges discovered during benchmark runs.
