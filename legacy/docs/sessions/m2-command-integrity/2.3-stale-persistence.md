# Session 2.3 — "Stale + audit trail persist"

## Objective

Complete the stale marker system and add event persistence to SQLite. Stale markers propagate through the dependency graph: marking an entity stale marks its dependents stale too (transitively). Clearing stale on a root entity clears it on dependents whose only stale cause was that root. Committed events are appended to a SQLite table for audit trail, diff view, and bundle export. After this session, the full stale lifecycle works end-to-end, and every canonical mutation is durably recorded.

## Behavior

### Stale propagation

- Rule: `mark_stale` appends a `StaleMarker` to the target entity's `stale_markers` array with `stale: true`, the given `reason`, `stale_since` as the current timestamp, and `caused_by` as the target entity's own ref.
- Rule: After marking the target, walk the inverse index to find all entities that reference the target. Mark each dependent stale with a reason like `"Dependency <type>:<id> is stale"` and `caused_by` pointing to the originally-stale entity (not the intermediate).
- Rule: Propagation is transitive — dependents of dependents are also marked.
- Rule: If a dependent already has a stale marker with the same `caused_by`, skip it (no duplicates). Multiple stale markers from different causes coexist in the array.
- Example: Marking `source_abc` stale propagates to `obs_1`, `obs_2` (which reference it), then to `claim_x` (which references `obs_1` via `based_on`).

### Stale clearing

- Rule: `clear_stale` removes the matching `StaleMarker` from the target entity's `stale_markers` array (filter by `caused_by`).
- Rule: For each dependent, remove the stale marker whose `caused_by` matches the cleared entity. The dependent remains stale if other markers remain in its array.
- Rule: If a dependent's stale marker was caused by a different entity, it is not cleared.
- Example: If `claim_x` is stale because `source_abc` was stale, and `source_abc` is cleared, then `claim_x` is also cleared (assuming no other stale causes).

### Stale behavior table

- Rule: Stale entities are always shown in views with a visual warning badge.
- Rule: Stale entities can be manually edited (stale never blocks editing).
- Rule: Stale entities drop solver readiness to `'usable_with_warnings'` or `'not_ready'`.
- Rule: Stale entities in scenarios get a "needs review" flag.
- Rule: Stale entities in play sessions are tagged "uses stale inputs."
- Rule: Stale markers are serialized into `.gta.json` (preserved across save/load).

### Event persistence

- Rule: After every successful `dispatch` that commits (not dry-run, not rejected), the resulting `ModelEvent` is appended to the SQLite `canonical_events` table.
- Rule: The persisted event log is append-only. Undo does not delete events from SQLite.
- Rule: On app launch, canonical state is loaded from `.gta.json`, not by replaying events. SQLite events are historical records, not the source of truth.
- Rule: The in-memory `EventLog` (for undo/redo) is rebuilt fresh each session. It does not load from SQLite.
- Rule: The audit trail must record every committed event. If a write fails, the system queues the event for retry. The Diff view and bundle export should warn when gaps exist in the audit trail.
- Rule: The persisted `canonical_revision` counter is the authoritative revision for conflict detection across restarts. The in-memory `EventLog.cursor` is used for undo/redo within a session, but `base_revision` on Commands is compared against the persisted revision. This ensures that play sessions (session 7.3) and AI proposals (session 5.3) that span restarts can still detect conflicts.

### Two event systems

- Rule: The in-memory `EventLog` (session 2.1) is for undo/redo only. It is session-scoped — rebuilt fresh on each app launch, lost on restart. It is NOT loaded from SQLite.
- Rule: The SQLite `canonical_events` table is for audit trail, diff view (session 3.5), and bundle export (session 8.1). It is workspace-scoped — survives restarts, append-only.
- Rule: These are separate systems with different lifecycles. Do not conflate them.
- Rule: The Diff view and bundle export query SQLite events. Undo/redo uses the in-memory EventLog.

### Integration with dispatch

- Rule: `mark_stale` and `clear_stale` commands flow through the normal dispatch cycle. They produce patches and inverse patches like any other command.
- Rule: The stale propagation logic runs as part of the integrity reactor (between reduce and commit), so all cascading stale changes are part of the atomic batch.

## Guidance

- The stale propagation walk is similar to the cascade walk in session 2.2 but with different semantics: cascade deletes entities, stale marks them. Reuse the inverse index traversal but do not conflate the two code paths.
- `caused_by` on `StaleMarker` always points to the root entity that was originally marked stale (by the user or AI), not the intermediate entity in the propagation chain. This makes `clear_stale` simple: clear everything whose `caused_by` matches the root being cleared.
- The SQLite integration can use `@tauri-apps/plugin-sql` in production. For tests, use an in-memory SQLite (`:memory:`) or a mock. Do not depend on Tauri being present at test time.
- The `PersistedEvent` stores `command_json` and `patches_json` as serialized strings. `ModelEvent.command` is a rich typed object — you need a serialization function. Since commands are designed to be serializable (for reproducibility bundles), this should be straightforward JSON stringify/parse, but test that discriminated unions round-trip correctly.
- Do not store `inverse_patches` in SQLite. They are only needed for in-memory undo and can be reconstructed if needed. This saves storage.
- SQLite event persistence failures should be treated as errors, not warnings. If the event cannot be written to the audit trail, the dispatch should still succeed (canonical state is the source of truth), BUT the failure must be surfaced to the user as a persistent warning banner: "Audit trail write failed — recent changes may not appear in history." The app should retry the write on next dispatch and log the gap. Silent data loss in the audit trail violates the auditability principle.

## Revision Notes (Step 2)

> Added 2026-03-15. M2 is already implemented. These notes describe stale/persistence changes needed for `RevalidationEvent` and phase progress tracking introduced in the M1 revision.

### RevalidationEvent dual storage

`RevalidationEvent` entities live in two places:

1. **L1 canonical (`.gta.json`)** — The `RevalidationEvent` entity is a first-class canonical entity like any other. It stores the trigger type, source phase, target phases, entity refs, description, and timestamp. It is created via the `trigger_revalidation` command (or `add_revalidation_event` for manual creation).

2. **SQLite audit trail (`canonical_events`)** — The `trigger_revalidation` command that created the entity is persisted as a `PersistedEvent` like any other committed command. This provides the audit trail entry.

No special dual-write logic is needed — the existing persistence pipeline handles both: the reducer puts the entity in the store (which serializes to `.gta.json`), and the post-commit hook persists the event to SQLite.

### `trigger_revalidation` stale propagation

When `trigger_revalidation` is dispatched:

1. The reducer creates the `RevalidationEvent` entity in the store.
2. For each `EntityRef` in the command's `entity_refs`, the integrity reactor dispatches `mark_stale` with reason: `"Revalidation triggered: <description>"` and `caused_by` pointing to the revalidation event entity.
3. The stale propagation from `propagateStale` then transitively marks all dependents of each affected entity.
4. All of this is folded into a single atomic batch — one undo reverts the entity creation and all stale markers.

### SQLite table for revalidation-specific queries

Add a new SQLite table (or extend existing views) to support efficient revalidation queries:

```typescript
// SQLite table: revalidation_log
interface RevalidationLogEntry {
  id: string                           // matches the RevalidationEvent entity ID
  analysis_id: string
  timestamp: string
  trigger_type: string                 // RevalidationTrigger discriminant
  source_phase: number
  target_phases_json: string           // serialized number[]
  entity_refs_json: string             // serialized EntityRef[]
  affected_count: number               // total entities marked stale (including transitive)
  description: string
}
```

This denormalized table enables queries like "show all revalidations that affected phase 3" or "list revalidations triggered by assumption changes" without deserializing every `canonical_events` row. It is populated in the same post-commit hook that writes to `canonical_events`.

### Phase progress tracking

Add a SQLite table to track analytical phase execution state:

```typescript
// SQLite table: phase_progress
interface PhaseProgress {
  analysis_id: string
  phase_number: number
  status: 'not_started' | 'in_progress' | 'completed' | 'stale'
  pass_number: number                  // how many times this phase has been run (starts at 0)
  last_completed_at: string | null
  stale_since: string | null           // set when a revalidation marks this phase stale
  stale_reason: string | null
}
```

**Behavior:**
- When a `trigger_revalidation` command targets phases via `target_phases`, the corresponding `phase_progress` rows are updated to `status: 'stale'` with the revalidation description as the reason.
- When a phase is re-executed (pass completed), `pass_number` increments, `status` returns to `'completed'`, and `stale_since`/`stale_reason` are cleared.
- Phase progress is L3 (derived) data — it lives only in SQLite, never in `.gta.json`. This is consistent with the data layer rules: SQLite holds derived/operational state, `.gta.json` holds canonical entities.
- The UI can query phase progress to show which phases need re-running after a revalidation event.

### Integration with existing stale system

The existing `propagateStale` and `clearStale` functions do not need structural changes. The new entity types participate in stale propagation automatically via the inverse index (which is driven by `REFERENCE_SCHEMA`). The only addition is that `trigger_revalidation` is a new entry point into the stale system — it calls `propagateStale` for each affected entity ref, which then follows the same transitive walk as any other `mark_stale` command.
