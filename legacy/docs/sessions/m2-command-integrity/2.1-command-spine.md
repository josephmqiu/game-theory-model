## Behavior

### Dispatch cycle

- Rule: Pre-validation checks the command structure (required fields present, referenced entity exists for updates/deletes, etc.).
- Rule: The reducer produces a candidate state by applying the command immutably (new object, never mutate the input store).
- Rule: The reducer also produces JSON Patch (RFC 6902) diffs and inverse diffs.
- Rule: The integrity reactor is called on the candidate state (stub in this session — session 2.2 adds real cascade logic). For now, it returns an empty array of `IntegrityAction`.
- Rule: Post-validation runs invariant checks on the candidate state (stub in this session).
- Rule: If all checks pass and `dryRun` is false, the event is pushed to the log, cursor advances, and the new store is returned.
- Rule: If `dryRun` is true, the event is returned without committing.
- Rule: If any check fails, nothing is committed and a `rejected` result is returned.

### Add commands

- Rule: `add_*` commands create a new entity. If `id` is provided, use it; otherwise, generate one via `generateEntityId`.
- Rule: Adding an entity with an ID that already exists in the store is a validation failure.
- Example: `{ kind: 'add_claim', payload: { statement: 'X', based_on: [], confidence: 0.8 } }` creates `{ id: 'claim_<uuid>', statement: 'X', based_on: [], confidence: 0.8 }` in `store.claims`.

### Update commands

- Rule: `update_*` commands merge the payload into the existing entity immutably. Only fields present in the payload are changed.
- Rule: Updating a non-existent entity ID is a validation failure.
- Example: `{ kind: 'update_claim', payload: { id: 'claim_abc', confidence: 0.9 } }` produces a new claim object with updated confidence, all other fields unchanged.

### Delete commands

- Rule: `delete_*` commands remove the entity from the store. The integrity reactor (session 2.2) will expand this into cascading actions.
- Rule: Deleting a non-existent entity ID is a validation failure.

### Batch commands

- Rule: `batch` commands apply all sub-commands as a single atomic unit. Either all succeed or none are committed.
- Rule: Undo/redo treats a batch as one unit — a single undo undoes the entire batch.

### Undo/redo

- Rule: Undo applies `inverse_patches` from the event at the current cursor, then decrements the cursor.
- Rule: Redo re-applies `patches` from the event after the current cursor, then increments the cursor.
- Rule: New commands after an undo truncate forward history (events after cursor are discarded).
- Rule: Undo at cursor 0 returns null (nothing to undo). Redo at cursor === events.length returns null.

### Revision protection

- Rule: If `base_revision` is set on a command and does not match the current cursor, the command is rejected with `revision_conflict` and a diff showing expected vs actual revision.
- Rule: If `base_revision` is not set, the command applies unconditionally (no conflict check).
- Rule: `base_revision` is compared against the persisted canonical revision counter (session 2.3), not the in-memory EventLog cursor. This ensures revision protection works across app restarts.

### Error recovery

- Rule: If the reducer throws an unexpected error, the candidate state is discarded, the store is not modified, and a `rejected` result with reason `'error'` is returned.

## Guidance

- The reducer must be a pure function: `(CanonicalStore, Command) -> { newStore, patches, inversePatches }`. No side effects, no store mutation. This is what makes undo/redo and dry-run work.
- Use a JSON Patch library (e.g., `fast-json-patch` or `immer` with patches enabled) to produce RFC 6902 patches. Do not hand-roll patch generation.
- The `Command` type has 48+ variants (3 per entity type times 16, plus structural commands). Use a code generation approach or a generic handler keyed on entity type to avoid 48 handwritten cases. The `STORE_KEY` mapping can drive this.
- `IntegrityAction` and `ImpactReport` are used in the `ModelEvent` and `DispatchResult` types. Define minimal stubs here (empty interfaces or type aliases) that session 2.2 will replace with full definitions.
- The dispatch function should NOT import or depend on Zustand. It operates on plain `CanonicalStore` and `EventLog` values. The Zustand integration happens in session 3.1.
- Never mutate the input `CanonicalStore` or `EventLog`. Always return new objects. This is a strict immutability requirement.
- Consider grouping the Command union hierarchically: `EntityCommand` (CRUD for all 16 entity types, driven by a generic `EntityCommand<K extends EntityType>` pattern using template literal types), `StructuralCommand` (update_payoff, mark_stale, clear_stale, remove_player_from_game), and `DomainCommand` (apply_cascade_effect, promote_play_result, batch). This keeps each sub-union to 10-15 members and improves TypeScript compilation performance. The top-level `Command` remains the union of these groups.

### New special command: `trigger_revalidation`

A domain-level command that creates a `RevalidationEvent`, marks affected entities stale, and logs to the audit trail. This is distinct from a simple `add_revalidation_event` because it has side effects beyond entity creation.

```typescript
| {
    kind: 'trigger_revalidation'
    payload: {
      trigger: RevalidationTrigger      // defined in M1 entity types
      source_phase: number
      target_phases: number[]
      entity_refs: EntityRef[]
      description: string
    }
  }
```

**Dispatch behavior for `trigger_revalidation`:**
1. Validate that all `entity_refs` exist in the store.
2. Create a new `RevalidationEvent` entity via the reducer (equivalent to `add_revalidation_event`).
3. For each entity in `entity_refs`, dispatch `mark_stale` with a reason linking to the revalidation event.
4. The stale propagation from session 2.3 then transitively marks dependents.
5. All of the above folds into a single atomic `batch` command internally — one undo reverts everything.
