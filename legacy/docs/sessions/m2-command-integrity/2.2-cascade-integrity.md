# Session 2.2 — "Deletes cascade correctly"

## Objective

Build the referential integrity layer that makes delete operations safe and predictable. Produce the full `REFERENCE_SCHEMA` declaring every cross-entity reference with its delete behavior, the `InverseIndex` for efficient dependent lookups, and the `ImpactReport` that shows what a delete will affect before it happens. After this session, deleting a `Source` cascades to its `Observation`s, marks dependent `Claim`s as stale when their `based_on` empties, and the integrity reactor folds all of this into the command's atomic batch.

## Contracts

### Types to create

```typescript
interface RefDeclaration {
  field: string
  target: EntityType | EntityType[]
  cardinality: 'one' | 'many'
  required: boolean
  on_delete: 'block' | 'remove_ref' | 'remove_ref_or_stale_if_empty' | 'cascade_delete' | 'mark_stale'
}

type IntegrityAction =
  | { kind: 'remove_ref'; entity: EntityRef; field: string; ref_id: string }
  | { kind: 'mark_stale'; entity: EntityRef; reason: string }
  | { kind: 'cascade_delete'; entity: EntityRef }
  | { kind: 'block'; reason: string }

interface ImpactReport {
  target: EntityRef
  direct_dependents: EntityRef[]
  transitive_dependents: EntityRef[]
  proposed_actions: IntegrityAction[]
  severity: 'safe' | 'warning' | 'destructive'
}

type InverseIndex = Record<string, ReadonlyArray<EntityRef>>
// Key format: refKey(targetRef), e.g., "claim:claim_abc123"
// Value: entities that hold a reference to the key entity
```

## Behavior

### Delete behavior per policy

- Rule: `cascade_delete` — the dependent entity is also deleted (recursively).
- Rule: `remove_ref` — the reference is removed from the field. The entity survives.
- Rule: `remove_ref_or_stale_if_empty` — the reference is removed. If this empties a `required: true` array field, the entity is marked stale instead of left with an empty required field.
- Rule: `mark_stale` — the dependent entity is marked stale with a reason referencing the deleted entity.
- Rule: `block` — the delete is rejected entirely. (Not currently used in the schema but the mechanism must exist.)

### Impact report

- Rule: `computeImpact` returns direct dependents (entities with a reference to the target) and transitive dependents (entities affected by cascading deletes of direct dependents).
- Rule: Severity is `'safe'` if only `remove_ref` actions, `'warning'` if any `mark_stale`, and `'destructive'` if any `cascade_delete`.
- Example: Deleting a `Source` produces: cascade-delete its `Observation`s, then for each deleted `Observation`, `remove_ref_or_stale_if_empty` on `Claim.based_on`. If a claim's `based_on` empties, the claim is marked stale.

### Inverse index

- Rule: The index is built once on load from the full `CanonicalStore`.
- Rule: The index is updated incrementally on each committed event (not rebuilt from scratch).
- Rule: For wildcard fields like `terminal_payoffs.*.source_claims`, the builder iterates all keys of the intermediate Record.

### Formalization subtype awareness

- Rule: `root_node_id` is only checked on formalizations with `kind: 'extensive_form'`. `stage_formalization_id` only on `kind: 'repeated'`. `agenda_setters` only on `kind: 'coalition'`. `parties` and `first_mover` only on `kind: 'bargaining'`. Missing fields on non-matching subtypes are ignored.

### Post-validation invariants

- Rule: After cascade expansion, verify: no dangling references remain, no duplicate IDs, no orphaned entities that should have parents, derivation edges form a DAG.
- Rule: **Chance probability sum** — All outgoing edges from a chance node (`actor.kind === 'nature'`) must have `chance_estimate.value` values that sum to 1.0 ± 0.001. Severity: error.
- Rule: **Information set action consistency** — All nodes in the same information set must have the same number of outgoing edges. Severity: error. Note: this invariant requires traversal into formalization subtypes (specifically `ExtensiveFormModel.information_sets`) — not just top-level entity references.
- Rule: **Independent probability warning** — When a `Scenario` has `probability_model: 'independent'`, return a warning: "Independent probability multiplication typically overestimates joint probabilities for real-world events. Consider dependency-aware mode." Severity: warning. (UI prominence decisions — persistent vs dismissible — belong in M3 view sessions, not here.)

## Guidance

- The `REFERENCE_SCHEMA` is the single source of truth for all cross-entity references. If a field holds an entity ID and it is not declared here, the integrity layer will not protect it. Be thorough.
- Wildcard field paths (`terminal_payoffs.*.source_claims`) require a field-path resolver that can walk into nested `Record<string, T>` objects. Write this as a utility function, not inline string parsing.
- The cascade walk must detect cycles to avoid infinite loops. Two entities referencing each other with `cascade_delete` would loop forever. Use a visited set.
- `expandCascade` returns a `batch` command. The dispatch function from session 2.1 already handles batch atomicity. Do not build a separate transaction mechanism.
- Do not modify the dispatch function's external API. Instead, integrate the integrity reactor as a hook that the dispatch function calls between reducing and committing. The dispatch function imports `expandCascade` and calls it when the command is a delete.
- Be careful with `remove_ref_or_stale_if_empty`: the "empty" check must happen after all refs in the same cascade batch are removed, not after each individual removal.

### Cascade behavior notes

- **`EscalationLadder`**: Uses `mark_stale` (not `cascade_delete`) on game deletion because a ladder is a standalone analytical artifact that retains value even if its parent game is restructured. The user should review and decide whether to keep or remove it.
- **`TrustAssessment`**: Uses `cascade_delete` for both player refs because a trust assessment is meaningless without both assessor and target.
- **`RepeatedGamePattern`**: Uses `cascade_delete` on game deletion because patterns are intrinsic to the game's history.
- **`CrossGameConstraintTable`**: Uses `remove_ref_or_stale_if_empty` on game deletion because a constraint table with fewer than 2 games is structurally invalid.
- **`RevalidationEvent.entity_refs`**: Uses `remove_ref` with a broad target list because revalidation events can reference any entity type. The revalidation event itself survives (it is an audit record), but stale references within it are cleaned up.
